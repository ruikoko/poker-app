## 1. pt50 — convergência independente da ordem + reset/re-import

### 1.1. Auditoria read-only (pedido do Rui)

Antes de mexer, auditoria de **5 pipelines** de ingestão à pergunta: *"a app é
independente da ORDEM de chegada dos inputs?"* (chave estável, idempotência,
placeholders, clobber). Veredicto: maioritariamente sim, com **3 dependências de
ordem reais** que se corrigiram com a BD vazia (momento ideal).

### 1.2. Os 3 fixes (`81255b2`)

- **#FIX-A (HM3 `all_players_actions`):** o UPDATE do PATH A do HM3 (`hm3.py`)
  podia **pisar** o `all_players_actions` já enriquecido (cruzamento SS↔HH) se a
  HH HM3 chegasse depois. Guard `CASE` no UPDATE: só escreve se o existente
  estiver vazio. *Porquê guard e não colapsar:* a união de tags do HM3 tem
  semântica própria; o que não se pode perder é o enriquecimento de nomes/acções.
- **#FIX-B1 (table-SS relink cobre ambíguos):** o relink órfão passou a
  reconsiderar também `tm_ambiguous`, não só `no_match_to_hand`.
- **#FIX-B2 (name-estrito):** validação de nome só nos sites de **nome fiável**
  (`_NAME_RELIABLE_SITES = {GGPoker, Winamax}`) — WPN/PS têm nomes genéricos
  (garantia / NULL), não servem para validar.

### 1.3. #FIX-B3 — match do table-SS desacoplado do upload (`6578f63`)

Refactor grande pedido pelo Rui. O match deixou de viver dentro do handler de
upload e passou a uma **função determinística R**:

- `compute_table_ss_match(ss, candidates)` — **puro** (sem efeitos): decide o
  match a partir da SS + candidatos.
- `_apply_hand_link(...)` — desliga link stale + liga o novo.
- `reconcile_table_ss()` — **re-avalia TODAS** as rows (incluindo `success`), não
  só as órfãs. Exposto em `POST /api/table-ss/reconcile`.
- `_persist_table_ss_match(...)` — escreve o resultado.

Ganho: idempotência total. Correr o reconcile N vezes converge sempre para o
mesmo estado, independentemente da ordem de chegada de SS e mãos.

## 2. pt51 — convenção de fuso UTC → LISBOA (`4645960`)

### 2.1. A ida-e-volta do fuso (contexto importante)

Houve **duas rotas**, em sessões diferentes:

1. **pt49 (rota anterior, JÁ no journal pt49):** `#GG-PLAYED-AT-LOCAL-NOT-UTC` —
   o parser GG (e PS) gravava `played_at` **verbatim** da hora local de Lisboa,
   sem normalizar. Fix pt49: converter GG+PS para **UTC DST-aware**
   (`lisbon_local_to_utc`), display Lisboa. Ficou pendente uma "validação de
   Inverno" (provar que a 1ª timestamp PS é DST-aware).
2. **pt51 (rota FINAL, este arco):** o Rui **mudou a convenção**: o ponto de
   encontro de **todas** as horas-de-evento passa a ser a **hora de LISBOA
   wall-clock**, guardada como **`timestamp` naive** (sem tz).

**Porquê Lisboa-naive em vez de UTC:** o Rui **joga sempre de Portugal**. A hora
útil para ele estudar é a hora a que jogou, em Lisboa. Guardar UTC obrigava a
reconverter em todo o lado (e a "validação de Inverno" só existia para provar a
matemática DST). Guardar Lisboa-naive **mata a ambiguidade DST na origem**: GG/PS
gravam-se **verbatim** (a string da HH já está em Lisboa), sem qualquer cálculo;
só as fontes que vêm em UTC (Winamax/WPN/Discord) é que se convertem **uma vez**
para Lisboa. A rota UTC foi assim **substituída** — ver TECH_DEBTS.

### 2.2. O que mudou (19 ficheiros)

- **Schema** (`schema.sql`): `played_at`, `discord_posted_at` (e companhia)
  `TIMESTAMPTZ → TIMESTAMP` naive. **Migração real** aplicada em prod (§B).
- **Helpers** (`utils/timezones.py`): `lisbon_naive_verbatim(dt)` (tira tzinfo),
  `utc_to_lisbon_naive(dt_utc)` (UTC → Lisboa, devolve naive).
- **GG/PS verbatim**: `gg_hands.py`, `hm3.py` (ramos PS/GG) gravam a hora da HH
  tal-e-qual (já é Lisboa).
- **UTC → Lisboa**: `tournament_summaries.py` (TS Winamax 16:00 UTC → 17:00
  Lisboa), `tournament_meta.py`, `attachments.py` (cast `::timestamp`),
  `discord.py`/`screenshot.py` (posted_at), `table_ss_vision.py` (captured_at).
- **Resolver** (`tournament_resolver.py`): anchor/janela deixaram de forçar tz —
  trabalham em naive.
- **Frontend** (`datetime.js`): `HAS_TZ` regex — ISO **sem** offset = Lisboa
  naive (mostra directo); ISO **com** offset (ex. `created_at` UTC) = converte.
- **19 testes reescritos** — codificavam a convenção UTC antiga.

### 4.1. Nome só-nome (`defd715`)

O Winamax mete sufixos no nome (`#NNN` = nº de mesa; `(ID)`). Helper
`clean_winamax_tournament_name(name) -> (clean, id)` apara `#NNN` + `(ID)` e

### 4.2. Desambiguação directa pelo nome (`901e965`)

Descoberto na verificação (caso GALACTICA): quando o table-SS tem **vários
candidatos** (multi_tn), em vez de delegar ao resolver de nomes frágil, compara o
**nome fiel da imagem** directamente com cada candidato (`name_tokens_subset`); se
**exactamente 1** `tournament_number` bate → liga (`disambiguated_by_name_direct`).
É este mecanismo que o pt58 reutiliza.

## 5. pt55 — `tools/appimport` (import por pasta local)

Ferramenta nova, estilo `apphm3`: agente Python standalone + `.bat` de
duplo-clique, **só a pedido** (salas fechadas, depois da sessão), nunca em
background. Routing **por subpasta** (determinístico):

| Subpasta | Endpoint | Destino |
|---|---|---|
| `gg_hh` | `POST /api/import` | `hands` |
| `gg_ts` | `POST /api/tournament-summaries/import` | `tournament_summaries` |
| `it` | `POST /api/table-ss/upload` | `players_left` + match |
| `manual` | `POST /api/screenshots` | enriquecimento/placeholder |

Incremental: ficheiro na subpasta = por enviar; sucesso (2xx) → **movido** para
`done/<tipo>/` (mover ≠ apagar); falha fica para retry. Re-enviar é seguro (a app
deduplica: HH por `hand_id`, SS de mesa por `file_hash`, TS por upsert). Auth =
cookie de sessão. `config_local.py` (gitignored) = pasta-mãe + credenciais.

## 6. pt56 — site do table-SS pelo nome do ficheiro

### 6.1. A razão de fundo (importante para operador novo)

O Vision do table-SS classificava o site errado (PS/WPN lidos como GG; o prompt
só descreve GG/WN). Esboçou-se um **prompt Vision multi-site (V2)** — e foi
**DESCARTADO**. Dois motivos:

1. **O feltro da mesa NÃO é sinal de site.** O Rui **muda o feltro por
   torneio/fase**, não por sala. Ensinar a Vision a ler o site pelo aspecto é
   ensinar ruído.
2. **As imagens do table-SS NÃO são guardadas** (descartadas após a Vision). Sem
   re-fornecer o ficheiro não há re-Vision — o que **bloqueia** validar um V2.

### 6.2. O fix determinístico (`45aa12a`)

O nome do ficheiro do Intuitive Tables traz o site literal:
`Shot<N>-<Site>-<YYYYMMDDHHMMSS>`. `_site_from_filename(filename)` lê o token [1]
(`_FILENAME_SITE_MAP = {GGPoker, Winamax, WPN, Stars→PokerStars}`), **autoritativo
sobre a Vision**. Token não-reconhecido (ex. `CoinPoker`) → `None` → fallback
Vision + log. *Decisão de engenharia:* **removi o `_correct_site` do
`compute_table_ss_match`** — para o site do filename não voltar a ser pisado pelo
cross-check BD no recompute.

por nome via manifesto `lobby_sent.txt`. `captured_at` = mtime do ficheiro
(âncora do torneio). Dois acabamentos pedidos pelo Rui antes de apontar à pasta:

- **`LOBBY_SINCE`** (`fc2ac7d`): só processa capturas com mtime ≥ data — evita

- **Retry de falha transitória** (`fc2ac7d`): um `vision_failed` (soluço de API)

  *Porquê:* o Rui apanhou que um soluço transitório devolvia 200/`is_lobby=False`
  e o agente marcava-o → **perdia um lobby real em silêncio**.

## 8. pt58 — tolerância a truncação do título GG (`eddca4d`)

O cliente GG **trunca** títulos longos no replayer (ex.
`268-M: $150 Saturday Secret KO [Mystery Bo...]`). O `name_tokens_subset` falhava
→ a SS (id=134) ficava órfã apesar de haver mão. Fix: se o **último token** vier
truncado (`...`), trata-se como **prefixo** (≥2 chars, substring); os tokens
**restantes mantêm-se exactos**. A unicidade fica garantida pelo caller (a
desambiguação directa pt54 exige **exactamente 1** candidato) → se o prefixo bate
2+ torneios, **fica ambíguo** (não inventa). **Não** se estendeu lógica de número
ao GG (títulos GG não têm ID). Recupera o id=134 → `disambiguated_by_name_direct`,
tn 286729346.
