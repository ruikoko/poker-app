### Bloco 1 — Commit E (sync-recent de lobbys + lobby_processing_log)

**Motivação:** dar ao Rui forma de re-processar SSs de `#lobbys` que
falharam ou nunca foram tentadas, e simultaneamente persistir cada
tentativa em BD (não depender de logs Railway).

**Decisões fechadas (D1-D8):**

| # | Decisão | Valor |
|---|---|---|
| D1 | Síncrono vs background | **Síncrono** (4-10 min worst-case, simples; UI mostra spinner) |
| D2 | Rate limit default | **1.2s** entre Vision calls (Tier 1 Anthropic ~50 RPM); param `vision_throttle_seconds` override no body |
| D3 | `max_messages` cap | **default 200, hard cap 500** |
| D4 | UI placement | sub-painel em `Discord.jsx`, abaixo do painel sync existente |
| D5 | Extracção do handler core | opção **α** — lógica para `services/lobby_sync.py:process_lobby_message`; handler real-time vira thin wrapper |
| D6 | Tabela `lobby_processing_log` (β) | ✅ **criada**; UPSERT por `discord_message_id` com `attempt_count++` em conflict |
| D7 | Reusar `tournament_resolver.resolve_tournament_number` | ✅ sim — TIER 0 já discrimina por prize_pool/total_players |
| D8 | Registar em `lobby_processing_log` no backoffice import | ❌ não — log é específico ao canal Discord; backoffice é upload manual |

**Schema novo (idempotente):**

```sql
CREATE TABLE IF NOT EXISTS lobby_processing_log (
    discord_message_id  TEXT PRIMARY KEY,
    channel_id          TEXT,
    attempted_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    attempt_count       INTEGER NOT NULL DEFAULT 1,
    result              TEXT NOT NULL,
    reason_detail       TEXT,
    site                TEXT,
    tournament_name     TEXT,
    tournament_number   TEXT,
    vision_json         JSONB,
    posted_at           TIMESTAMPTZ
);
```

**Refactor:**

- `_handle_lobby_message` (real-time) reduzido de ~180 linhas inline para
  ~95 linhas wrapper. Texto exacto das replies preservado para não partir
  os tests existentes (`test_discord_lobby_handler.py`).
- `process_lobby_message` em `services/lobby_sync.py` é agnóstica de
  `discord.py` (recebe bytes/mime, devolve `{result, candidates,
  bounty_type, progressive_factor, action, …}`).
- Imports por módulo (`from app.services import lobby_vision, …`) para que
  patches existentes em tests apanhem os usos por via dinâmica.
- `_upsert_lobby_log` defensivo — falhas BD viram `logger.error`, não
  partem o handler real-time.

**Endpoint:** `POST /api/lobbys/sync-recent` com body Pydantic
(`since/until/sites/max_messages/dry_run/vision_throttle_seconds/
failure_types/retry_success`). RuntimeError → 503/404.

**Selecção de candidatos:** cruza `channel.history(since..until)` com
`lobby_processing_log` — mensagens sem `result='success'` (ou todas se
`retry_success=true`), filtradas opcionalmente por `failure_types`.

### Bloco 3 — Operações ad-hoc INSERT manual

(tn 283542120) ao primeiro pedido do Web. **Era o tn errado** — Rui jogou

**Correcção transaccional num só script** (`_local_only/_q_pt20_correction_283542054.py`):

1. `DELETE FROM tournament_payouts WHERE tn = '283542120'` → 1 row apagada ✓
2. `INSERT … tn = '283542054'` com payouts correctos →
   `action=inserted, n_prizes=18, sum=9420.80` ✓

**Lição:** verificação cruzada (Rui ↔ Web ↔ Code) antes de INSERT em prod
funcionou — Web apanhou o tn errado num passo de revisão.

| D13 | Mystery in-scope? | ❌ — `mystery_unsupported` fail-fast; tech debt `#BACKOFFICE-MYSTERY` |

**Pipeline:**

```
upload (1 img ou .zip)
  ↓
loop por imagem:
  detect_image_mime(bytes)                      # services/image_utils
  raw = extract_backoffice_payout_json(...)     # Anthropic Sonnet 4.6
  parse_and_validate (1ª pass, sem ratio)
    vanilla: |sum(prize) - pool| <= 0.05
    PKO: devolve sentinela _error='missing_pko_ratio'
  resolver.resolve_tournament_number(...)       # TIER 0 prize_pool+players
  _lookup_ts_meta(tn) → tournament_format, tournament_pko_ratio
  if format == 'KO' → result='mystery_unsupported' (fail-fast)
  PKO: 2ª pass com ratio → validação 2% drift; warning ss_likely_truncated
  blob = build_backoffice_payouts_blob(vj, format, ratio)
  upsert_payout(source='backoffice_vision:<fn>')
```
