## §PROCESSO — pontos sem rasto no git

### 1. Origem do fix WPN
Surgiu da **exploração do HRC watcher**: ao olhar a pipeline percebeu-se que o **WPN
não estava no `ALLOWED_SITES`** de `hrc_queue.py` — logo nenhuma mão WPN chegava à
fila HRC, independentemente da tag.

### 2. Gate WPN por FORMATO — tentado e ABANDONADO
A 1ª abordagem foi gatear a WPN **por formato** (`site<>'WPN' OR tournament_format
∉ {pko, super ko, ko}`), reusando o `detect_tournament_format` (por nome). **Tinha
leak**: a HH WPN **não traz marcador de bounty** (sem sinal estrutural na mão), por
isso um WPN bounty com nome "limpo" — ex.: `$10,000 GTD` que é PKO — seria detectado
como **Vanilla** e entraria na fila como non-KO. Risco real. **Substituído pelo gate
por TAG** (`#WPN-ICM-TAG-GATE`): a **classificação do Rui** (tag ICM/ICM FT) é o
*ground truth*, não o nome do torneio. O código por-formato chegou a estar na working
tree (suite verde) mas **não foi committado** — foi descartado antes do commit.

### 3. Confirmado: o HRC só recebe PRÉ-FLOP
O HRC modela só o pré-flop. Isto está **garantido pelo basket** de tags do export
(`DEFAULT_TAGS` só tem tags de pré-flop: `icm-pko`, `ICM`, `ICM FT`, `PKO SS`,
`sqz-pko`, `speed-racer*`), e o gate WPN segue a mesma regra (`{ICM, ICM FT}`). As
tags **pós-flop** (`pos-pko`, `pos-nko`, …) **não entram** — vale para **todas as
salas**, não só a WPN. Por isso `pos-*` ficar de fora do gate WPN é correcto.

### 4. Estado do filename tn novo do Intuitive Tables
- **WN / GG / PS:** formato novo (com tn) **confirmado**.
- **WPN:** **por confirmar** — falta ver uma SS WPN no formato novo (traz tn?). O
  **fallback protege** (sem tn no nome → `filename_tn=None` → fluxo actual: Vision +
  resolver). Zero risco enquanto não se confirma.
