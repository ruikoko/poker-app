## Bloco 2 fix urgente — `#META-AGGRESSOR-POSITION` (15 Maio, este commit)

Extensão de `derive_aggressor_real_action` com campo `position` (string maiúsculas — labels canónicos de `_POSITION_LABELS_BY_N`: UTG/HJ/CO/BTN/SB/BB/EP/MP/EP1/EP2 + `BU/SB` para HU). Mapping nick → position via `derive_seats_in_preflop_order` (única fonte de verdade do preflop order pt25d) — qualquer mudança de convenção futura fica centralizada lá.

### Schema final

```json
{
  "type": "raise" | "bet",
  "size_bb": float,
  "position": "UTG" | "HJ" | "CO" | "BTN" | "SB" | "BB" | "EP" | "MP" | "EP1" | "EP2" | "BU/SB" | null
}
```

`position = None` apenas em casos degenerate (parsing de seats falha, nick do raiser não está nos seats — implica HH inválida). Em condições normais cross-site (PS/GG/WN/WPN) a position é sempre derivável.

### Helper privado `_resolve_position_for_nick`

```python
def _resolve_position_for_nick(hh_text: str, nick: str) -> Optional[str]:
    if not nick:
        return None
    for s in derive_seats_in_preflop_order(hh_text):
        if s.get("nick") == nick:
            pos = s.get("position")
            return pos if isinstance(pos, str) else None
    return None
```

Reutiliza `derive_seats_in_preflop_order` (pt25d). Não duplica lógica de preflop order — qualquer fix futuro na convenção de índices ou labels propaga automaticamente.

### Injecção

Campo `position` viaja nos mesmos sítios que `type/size_bb` (deployed em Bloco 1):
- `manifest.json` → `hands_included[*].aggressor_real_action.position`
- `<hand_id>/payouts.json` → `aggressor_real_action.position`

Sem schema change disruptivo. Forward-compat com o `.exe` pt25d (que ignora `aggressor_real_action` por completo). Forward-compat com Bloco 2 wiring (que vai ler `position` para o OCR match).
