Cada fix expôs o bug seguinte a jusante:

1. **Detectar o fim de uma run correctamente** (pt30 estado do Finish,
   pt31 janela de progresso) →
2. **Fazer o click do Play da 2ª run pegar** (pt32 v1 coord Y, pt32 v2
   origem `find_hrc()`) →
3. **Fazer o OK do popup Nash pegar** (pt33 BM_CLICK) →
4. **Esperar a 2ª run pelo título certo** (pt34 substring "Monte Carlo
   Sampling").

No fim, a 2ª run em Selected Subtree funciona como o Save As manual.

## pt30 — `52aef9c` — polling Win32 do botão Finish

- **Sintoma:** o smoke pt29-v3 falhava. O slow-click no Finish (pt29-v2,
  validado) parecia não pegar de novo em certas mãos.
- **Hipótese inicial:** regressão do slow-click ou do foco.
- **Investigação:** script de diagnóstico `check_wizard_children_polling`
  enumerou as child windows do wizard "Hand Setup". **Descoberta-chave:**
  o HRC usa **SWT** (não Swing) e expõe os widgets como child windows
  nativas ao Win32 — o Finish aparece como `class='Button'`, `text='&Finish'`,
  e `IsWindowEnabled` é fiel. Mediu-se que, ao carregar o script, o HRC
  dispara o cálculo do tree size e **desabilita** o Finish até terminar
  (~1.7s na mão de teste). O slow-click caía num botão **disabled** —
  causa real do pt29-v3 falhar.
- **Decisão:** antes do slow-click, esperar a transição
  `enabled → disabled → enabled` do Finish via Win32.
- **Fix:** instância **isolada** `_pt30_user32 = ctypes.WinDLL("user32")`
  (para não colidir argtypes com o `EnumChildWindows` do launcher Baltazar,
  que usa um callback de assinatura diferente) + argtypes/restype
  explícitos (obrigatório em 64-bit). Helpers `_is_enabled`,
  `_resolve_wizard_hwnd`, `_find_finish_button`, `_wait_for_finish_ready`.
  Fase 1 (5s) → disabled (calc arrancou; WARN graceful se não); fase 2
  (60s) → enabled (calc terminou; raise se preso). Inserido em `setup_hand`
  entre `setup_scripting` e o slow-click (que fica intacto).
- **Resultado do smoke:** Finish passa a pegar de forma fiável.
- **Discovery:** **SWT, não Swing.** Este facto desbloqueou todo o resto
  da sessão — o popup Nash (pt33) e as janelas de progresso (pt31/pt34)
  são todos atacados pela mesma via Win32.
- **Testes:** +6 (5 unitários + 1 wiring). Suite 550→556.

## pt31 — `0f159bc` — polling da janela de progresso "Hand Setup"

- **Sintoma:** o smoke pt30 mostrou que a heurística de memória
  `wait_for_calculation` (Baltazar OG, instalada em pt29-v3) dava **falso
  positivo**: declarava o fim da 1ª run aos 48s (= 15s sleep inicial +
  3 ciclos × 10s), mas a run real ainda corria há muito mais tempo.
- **Hipótese:** a heurística "memória estável" não é fiável para árvores
  grandes — a memória estabiliza transitoriamente enquanto o cálculo ainda
  decorre.
- **Investigação:** confirmou-se que existe uma janela top-level de
  **progresso** com título "Hand Setup" (visualmente distinta do wizard,
  mutuamente exclusiva no tempo) que está presente enquanto a run corre e
  desaparece quando termina. Sinal binário, sem heurística.
- **Decisão:** substituir a heurística de memória por polling Win32 dessa
  janela.
- **Fix:** `_wait_for_run_completion(timeout_appear_s=30,
  timeout_total_s, run_label, ...)`. Fase 1 aguarda a janela aparecer
  (run arrancou; WARN graceful se não — run trivial/erro). Fase 2 polla
  até desaparecer (run terminou; log minuto-a-minuto; raise no timeout).
  Usa o mesmo `_pt30_user32` isolado do pt30. Constantes
  `_RUN_WAIT_POLL_S=0.5`, `_RUN_WAIT_PROGRESS_LOG_S=60.0`. Timeouts: 2h
  (1ª run, ~10M iterações), 8h (2ª run, pode demorar horas).
  `wait_for_calculation` fica no namespace mas já não é chamada.
- **Resultado do smoke:** a 1ª run passa a ser esperada até ao fim real.
- **Testes:** +4 (normal, never-appears graceful, never-disappears raise,
  wiring). Suite 556→560.

## pt32 v1 — `61dfa5f` — coord Y 59→64 + logging `[calc-diag pre-click]`

- **Sintoma:** o smoke pt31 confirmou visualmente que o click do Play da
  2ª run **não pega** — o popup "Nash Calculation" nunca abre,
  `_wait_for_nash_popup` dá timeout.
- **Hipótese:** coord do botão Play errada.
- **Investigação (read-only):** a 1ª run (Baltazar OG, `start_calculation`)
  usa `(204, 64)` e funciona; a nossa 2ª run usava `(204, 59)` — 5px off.
- **Decisão:** alinhar a coord Y com a 1ª run + adicionar logging para o
  caso de não bastar.
- **Fix:** `CALCULATE_BUTTON_Y` 59→64. Logging `[calc-diag pre-click]`
  (coord absoluta + foreground via `GetForegroundWindow`), em paralelo
  ao `[finish-diag pre-click]` do pt30.
- **Resultado do smoke:** **falhou** — o popup continuou a não abrir. Mas
  o logging foi decisivo: imprimiu `coord=(1174,64)` com `wpos=(970,0,...)`,
  o que desbloqueou o diagnóstico do pt32 v2.
- **Testes:** +3. Suite 560→562.

## pt32 v2 — `c9c8818` — origem do click: `wpos` → `find_hrc()`

- **Sintoma:** popup Nash continua a não abrir após pt32 v1.
- **Investigação:** o log do pt32 v1 dizia tudo. `coord=(1174,64)` com
  `wpos=(970,0,...)` → 1174 = 970 + 204. Ou seja, `_click_calculate_button`
  usava `wpos` como origem — e `wpos` era a geometria do **wizard "Hand
  Setup", já fechado pelo Finish da 1ª run**. O HRC estava em foco, mas o
  click caía em zona vazia.
- **Achado de raciocínio (Rui):** o Rui apanhou o erro do Web. A observação
  dele — *"o cursor estava em cima do Play durante a 1ª run, porque é que
  não copiamos essa abordagem?"* — foi o que desbloqueou o fix. A 1ª run do
  Baltazar OG usa `find_hrc()` como origem; a nossa 2ª run estava a inventar
  uma origem a partir de uma janela já fechada.
- **Decisão:** alinhar a 2ª run com a 1ª — origem = `find_hrc()`.
- **Fix:** `_click_calculate_button` passa a calcular
  `(hrc.left + 204, hrc.top + 64)` via `find_hrc()` e a clicar com
  `pyautogui.click` (deixa de usar `click_rel`). `find_hrc()` None → WARN +
  raise (não no-op silencioso). `wpos` mantém-se na assinatura mas é
  ignorado. Logging enriquecido com a janela HRC encontrada.
- **Resultado do smoke:** **sucesso — o popup Nash abre.**
- **Testes:** +1 (raise quando find_hrc None). Suite 562→563.

## pt33 v1 — `867460c` — OK do popup Nash via BM_CLICK Win32

- **Sintoma:** o smoke pt32 v2 confirmou que o popup Nash abre e que os
  cliques intra-popup (dropdown scope, opção Selected Subtree, campo CI +
  escrita) funcionam visualmente. Mas o **OK não pega** — o popup fica
  aberto e parado, e a 2ª run não dispara realmente. `Tab`/`Enter` não
  funcionam dentro do popup (confirmação visual do Rui).
- **Hipótese:** o `pyautogui.press('enter')` não chega ao botão OK do popup.
- **Investigação:** snapshot Win32 do popup
  (`_local_only/diag/check_nash_popup_children.py`, via EnumChildWindows).
  **Anatomia descoberta:** o popup é um dialog `#32770` top-level com 6
  `SWT_Window0` (containers internos, **sem** widgets expostos para o
  dropdown de scope e campo CI) e **2 `Button` nativos expostos** (`OK`,
  `Cancel`) — à imagem do wizard "Hand Setup" do pt30. Implicação: scope/CI
  continuam dependentes de coord, mas OK/Cancel podem ir via BM_CLICK.
- **Decisão:** substituir o OK por Enter por enumeração + `BM_CLICK` no
  hwnd do Button OK, análogo ao Save btn do export.
- **Fix:** `BM_CLICK = 0x00F5` + `SendMessageW` no `_pt30_user32`. Helpers
  `_find_nash_popup_hwnd` (FindWindowW + EnumWindows substring) e
  `_find_ok_button` (EnumChildWindows class='Button' text normalizado=='ok').
  `_click_ok_in_popup` reescrito para BM_CLICK determinístico (sem coord,
  sem foco) + WARN se popup ou Button OK não encontrados (sem fallback
  Enter — failure explícito > Enter silencioso que já provou não funcionar).
- **Resultado do smoke:** **sucesso — o popup fecha, a 2ª run dispara.**
- **Testes:** +3 líquido (−1 teste Enter, +4 BM_CLICK/WARN/guard).
  Suite 563→566.

## pt34 v1 — `e58c517` — espera da 2ª run por substring "Monte Carlo Sampling"

- **Sintoma:** o smoke pt33 v1 confirmou que o BM_CLICK no OK funciona — a
  2ª run arranca a sério. Mas `_wait_for_run_completion` da 2ª run dava
  **timeout 30s** na fase "esperar a janela aparecer".
- **Investigação:** a janela de progresso da 2ª run não se chama "Hand
  Setup" — chama-se **"H-\<hand_id\>: Monte Carlo Sampling"** (confirmação
  visual no Beelink). A fase 1 procurava título exacto "Hand Setup", nunca
  encontrava, e o robot avançava para o Save As com a 2ª run ainda a correr.
- **Decisão:** distinguir 1ª vs 2ª run no critério de match do título.
- **Fix:** param `match_substring` em `_wait_for_run_completion` + helper
  `_find_progress_window_title` (None → `FindWindowW` exacto "Hand Setup";
  preenchido → `EnumWindows` substring case-insensitive). A 2ª run passa a
  procurar "Monte Carlo Sampling"; a **1ª run fica inalterada** (título
  "Hand Setup" funciona como antes). Log enriquecido com o título completo
  encontrado.
- **Resultado do smoke:** **sucesso ponta-a-ponta.** A 2ª run espera os
  ~14 minutos completos em vez de timeout 30s. O export final saiu com um
  `.zip` de **~23 000 nós**, equivalente ao Save As manual da sessão
  anterior.
- **Testes:** +3 (substring match, exact match regressão, passthrough
  2ª run). Suite 566→569.

## Achados operacionais transversais da sessão

- **SWT, não Swing.** O HRC expõe os seus widgets (botões do wizard,
  botões do popup Nash) como child windows nativas ao Win32. Isto permite
  `BM_CLICK`, `IsWindowEnabled`, `GetWindowText` — toda a sessão assenta
  nisto. Detalhe em `HRC_ANATOMIA_OPERACIONAL.md` §1/§6/§9.
- **Popup Nash `#32770`:** dropdown de scope e campo CI **não** têm
  widgets Win32 expostos (vivem dentro dos `SWT_Window0`), logo continuam
  a depender de coordenadas; OK/Cancel são `Button` nativos → BM_CLICK.
- **Atalhos de teclado não funcionam dentro do popup Nash** (confirmação
  visual do Rui). `&Finish` (Alt+F) e `Alt+R` (Play) só funcionam quando
  o foco está no painel principal das estratégias.
- **Cursor anomaly pós-Save-As:** o Rui observou que, após o Save As, o
  cursor da Strategy Table fica na **2ª linha (EP)**. Origem desconhecida.
  Não bloqueia, mas merece registo (`#CURSOR-ANOMALY-POST-SAVE-AS`).
