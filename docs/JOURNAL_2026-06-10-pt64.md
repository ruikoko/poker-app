# JOURNAL pt64 — escrita do Scope da 2ª run via SysListView32 do dropdown (10 Junho 2026)

## Problema
Na re-smoke pt61b o Scope da 2ª run não passava a "Selected Subtree": a via Win32
de pt61 (`_find_combo_with_item` → `CB_SETCURSEL`) devolvia `(None,None)` → o
read-back nunca confirmava → `[ABORT]` (correcto) → a fila avançava com a mão a
`.failed`. Igual nas 2 mãos do smoke (GG-6027196507 e WN-…-1780348385). Tudo o
resto validado: navigate acertou o nó, finish/wizard/1ª run OK, cadeia
falha→abort→avanço impecável. O Rui mudou o Scope À MÃO no mesmo popup e a 2ª run
completou — logo o controlo é operável, só não pela via Win32 CB_*.

## Diagnóstico (2 diags no Beelink)
- **diag-1** (`diag_nash_popup.py`): o popup Nash (`#32770`) é um painel SWT opaco —
  `SWT_Window0` aninhados, todos `patterns=[]`, sem `ComboBox`/`Edit` nativo nem UIA;
  só `OK`/`Cancel` são nativos. **Caso 3** (canvas pintado). Nem CB_* nem UIA pegam
  no Scope.
- **diag-2** (modo `dropdown`): ao abrir, o CCombo do Scope **cria uma janela
  top-level nova** (`#32770`, ~250×40) com uma **`SysListView32` NATIVA** cujos itens
  o UIA lê pelo Name: `Full Tree` (idx 0) e `Selected Subtree` (idx 1). → **Via (A)**,
  a melhor: controlo real, endereçável, sem comtypes.

## Fix (`tools/watcher_src/patched_funcs.py`, só `_set_scope_in_popup`)
Imitação humana + read-back barato por mensagens de ListView (int, **sem** buffer
cross-process, **sem** comtypes — zero deps novas no PyInstaller):
1. foreground do popup + foco-click no campo Scope (rel pt61 `(308,69)`);
2. abrir o dropdown por `F4` (retry 1×);
3. confirmar a abertura achando a `SysListView32` do dropdown
   (`_find_scope_dropdown_listview`: top-level `#32770` que contém uma SysListView32);
4. seleccionar "Selected Subtree": type-search `"Selected"` (text-driven) com
   fallback determinístico `Home`→`Down` (Full Tree 0 → 1);
5. **read-back**: `LVM_GETNEXTITEM(LVNI_SELECTED)` confirma idx==1 (≠ default 0),
   numa lista de exactamente 2 itens, **antes** do commit;
6. `Enter` confirma o item destacado.

Sem confirmação no passo 5 → `Esc` + `return False` → o caller **ABORTA** a 2ª run
(salvaguarda/avanço **intactos**, validados em pt61). Gate `count==2`: se a lista
mudar (novo item/reordenação), aborta com WARN (re-correr o diag e actualizar o mapa
`SCOPE_LIST_SELECTED_SUBTREE_INDEX`). Helpers novos: `_find_scope_dropdown_listview`,
`_lv_item_count`, `_lv_selected_index`. Constantes novas: `LVM_*`, `LVNI_SELECTED`,
`SCOPE_LIST_SELECTED_SUBTREE_INDEX=1`. O caminho CB_* de pt61 fica morto (o CI continua
no seu caminho próprio — é o mesmo Caso 3, tratamento deferido p/ pt65).
