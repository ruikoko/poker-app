### Bloco 5 — PASSO 5b: re-bundle PyInstaller via trampoline

**Estratégia escolhida:** wrapper.py finíssimo (49 linhas) que `marshal.load` o launcher Apr19 intacto e exec. Razão: pycdc só recuperou 5/12 funções do launcher; reescrever `make_patched_export`/`_find_ok_button`/etc. seria perigoso.

**3 fixes intermédios descobertos via smoke standalone:**
1. **Hangup pipe deadlock** no supervisor Python (subprocess.Popen com PIPE sem leitura activa enche o buffer). Mudança para `Start-Process` + file redirect.

### Bloco 7 — PASSO 7: hotfix Bug E (MTT Stacks page com Multi Table ICM)

### Bloco 8 — PASSO 8: adapter detect_done_zips fix

**Sintoma:** após PASSO 7 .exe novo no Beelink, smoke gera zip em `done\Exports\GG-5914506215.zip` (3.3 KB), adapter faz `startup_scan: nada pendente` e `tick: 0 nova(s) puxada(s)` — zip órfão invisível.

**Causa raiz:** `detect_done_zips` faz `done_dir.glob("*.zip")` não-recursivo, esperando `done\<hand>.zip` directo. Watcher Apr19 grava em `done\Exports\<hand>.zip` (subpasta).

**Fix:** path explícito (mais defensivo que `rglob` que apanharia zips em subpastas inesperadas):

```python
zips.extend(done_dir.glob("*.zip"))           # backward-compat layout flat
exports_dir = done_dir / "Exports"
if exports_dir.is_dir():
    zips.extend(exports_dir.glob("*.zip"))    # layout Baltazar Apr19
```

Adapter PC principal editado, instruções de deploy ao Beelink entregues. Sem mexer no watcher.

### Bloco 9 — PASSO 9: adapter meta.json inject

**Sintoma:** após PASSO 8 fix activo no Beelink, adapter detecta zip e faz POST → `HTTP 400 {"detail":"meta.json missing in zip"}`.

**Causa raiz:** zip HRC nativo contém só `settings.json` + `equity.json` + `nodes/0.json`. Sem `meta.json`. Baltazar designou um "bot externo" que movia zip de `done/Exports/` → `done/replied/` e `inject_meta_into_zip` no main loop adicionava `meta.json` com `{rank, players_left, stage, ci}` lidos de `queue/<hand>/meta.json`. Esse bot **não existe na pipeline poker-app→adapter→watcher**.

**Fix (Opção A — adapter injecta):** novo helper `_ensure_meta_in_zip(zip_path, hand_id)` chamado em `post_done` antes do POST. Idempotente: zips já com meta.json ficam intactos. Caso contrário repacka adicionando minimal `{hand_id, exported_at, source: "hrc_adapter_inject", watcher_built_meta: false}`. Backend depois augmenta server-side com `{received_at, received_from}`.

Smoke isolado 3/3 PASS (caso sem meta, caso com meta, entries preservadas).

Razão de escolher (a) vs (b) backend tolerant: sem backend deploy adicional; protocolo POST mantém-se íntegro; adapter é a fronteira natural entre Beelink e backend. Tech debt `#WATCHER-META-INJECTION-BYPASSED` LOW para refactor pt24+ remove dead code do Baltazar.
