**Causa raiz identificada via leitura do código + análise read committed isolation:**

`backend/app/routers/hm3.py:728-855` (`_create_hand_villains_hm3`), linhas 762-772:
```python
hand_meta = {}
from app.db import query as _q
rows = _q(
    """SELECT hm3_tags, discord_tags, has_showdown,
              player_names->>'match_method' AS match_method
       FROM hands WHERE id = %s""",
    (hand_db_id,)
)
if rows:
    hand_meta = dict(rows[0])
```

`_q` = `app.db.query` que abre **CONEXÃO SEPARADA** (`backend/app/db.py:35-39`):
```python
def query(sql, params=None):
    with get_conn() as conn:           # NOVA conexão
        with conn.cursor() as cur:
            cur.execute(sql, params)
            return cur.fetchall()
```

A função `_create_hand_villains_hm3` é chamada DENTRO da mesma transação do INSERT em hands (`cur` partilhado, `conn.commit()` só na linha 1197 depois do for-loop terminar).

**PostgreSQL read committed isolation:** uma conexão só vê dados de outras transações **após COMMIT**. A hand recém-inserida ainda não está committed → `_q` (conexão separada) recebe `rows = []` → `hand_meta = {}` → no helper:
- `hm3_tags = []` → regra A não dispara
- `match_method = None` → regra B/C não dispara
- `cats = []` → INSERT skipped silenciosamente

Por isso 159 hands criadas com 0 entries em `hand_villains`.

#### Fix Tech Debt #9 (commit `630dc73`)

Substituir `_q()` (conexão separada) por:
- `hm3.py:763-772` → `cur.execute()` directo (`cur` já é parâmetro da função, transação activa)
- `mtt.py:660-668` → `with conn.cursor() as _cur` dentro de `if hand_db_id` (cursor temporário na mesma `conn`, mesma transação, preventivo — `mtt.py` tinha o mesmo padrão buggy mas call-sites Discord/SS comitam antes, daí não se manifestar em prod)

#### Issue 1 — Timezone display

Rui detectou: BD reporta hand 5 played_at = '2026-04-27 20:42:48' mas app UI mostra hora mais tarde.

Investigação:
- Schema: `hands.played_at` é `timestamp with time zone` (timestamptz)
- Hand 5 stored: `2026-04-27 20:42:48+00:00` (UTC, server timezone `Etc/UTC`)
- Raw HH Winamax tem literalmente `2026/04/27 20:42:48 UTC` no header
- Parser HM3 (`hm3.py:298-306`) cria `datetime` naive sem tzinfo → server interpreta como UTC ✅ correcto
- Frontend `HandRow.jsx:131-139` faz `new Date(hand.played_at).getHours()` → hora local do browser (Lisbon UTC+1 em DST Verão)
- Comentário explícito no código: "DD/MM + HH:MM em TZ local. Antes slicava a string ISO (UTC), agora converte via Date para bater com as horas a que o utilizador jogou."

**Conclusão: NÃO é bug.** BD em UTC (canónico) + UI em Lisbon (conveniente). Diferença = +1h em DST Verão WEST. Registado como **nota explicativa em §7.4** para evitar futura confusão.

## Decisões arquitecturais importantes

### Verificação orgânica vs SQL ad-hoc

A regressão `hand_villains=0` só foi apanhada porque a validação foi **end-to-end via fluxo real** (`.bat` HM3 a inserir mãos como em produção), não via backfill SQL ad-hoc (que foi como o Tech Debt #4 Parte B foi validado na sessão 28-Abr — daí o bug latente passar despercebido).

### Wipe TOTAL + repopulação real vs backfill SQL

Wipe + repopulação valida o **caminho de escrita inteiro**, incluindo tudo o que acontece dentro de transações backend. SQL ad-hoc só valida o estado final, escapa bugs cross-connection / order-of-operations.

### Fix mínimo + preventivo

Tech Debt #9 corrigiu o caso confirmado (`hm3.py`) + caso paralelo (`mtt.py:_create_villains_for_hand`) preventivo. Bug latente em `mtt.py:786` deixado como pendente registado para limitar scope (não afecta prod).

### Spec rigorosa antes de código (Tech Debt #12)

A re-arquitectura modal Vilões teve spec exaustiva via Q&A iterativo (layout, lista colapsada, mão expandida, notação compacta de acções, cards villain only, BB delta) ANTES de qualquer linha de código. Estimativa 0 retrabalho na implementação futura.

## Lições

1. **Verificação ponta-a-ponta orgânica é o único método que apanha bugs cross-connection.** Backfill SQL ad-hoc valida o estado final mas escapa bugs no caminho de escrita (read committed isolation, transações encapsuladas, conexões separadas).

2. **Wipe + repopulação real > restore parcial.** Mesmo com backup defensivo de 80 MB, o ciclo wipe-import-validar é mais limpo e mais rápido que tentar reproduzir o bug em ambiente isolado.

3. **`_q` (conexão separada) é code smell em transações activas.** Toda a chamada `from app.db import query as _q` dentro de uma função que recebe `cur` ou `conn` deveria usar a transação activa em vez. Pattern a banir em revisões futuras.

4. **Spec rigorosa via mockup + Q&A iterativo poupa retrabalho.** Tech Debt #12 (re-arquitectura modal Vilões) gerou ~30 perguntas estruturadas com respostas A/B/C. 0 ambiguidade restante; implementação futura é pura execução.

5. **"Não é bug" é um resultado válido de investigação.** Issue 1 timezone foi investigado a fundo e fechado como nota explicativa. Documentar comportamentos correctos não-óbvios evita futura re-investigação.
