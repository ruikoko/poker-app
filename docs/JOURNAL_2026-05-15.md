  - `GET /api/hrc/sessions/:id` (detalhe + settings + equity).
  - `GET /api/hrc/sessions/:id/nodes/:idx` (node completo com hands_strategies).

**Frontend** (`pages/HRCSessions.jsx` + `pages/HRCSessionDetail.jsx` + nav item "HRC" no Shell):

## Noite — `#ORFA-HM3-SYNTHETIC-ENTRIES`

### Investigação

Rui apanha visualmente que mãos HM3 (das fontes Winamax/WPN/PokerStars do `.bat`) aparecem com badge "Órfã" no UI. Investigação revela: o importer HM3 antigo escreve directo na tabela `hands` **sem** passar por `entries` → `entry_id` fica NULL → o painel "Inbox" usa `entry_id IS NULL` como heurística para Órfã → 1117 mãos HM3 fantasma. Discord/screenshot uploads passam por `entries` há muito tempo; HM3 ficou fora desse modelo durante meses.

Decisão: criar uma `source` nova `'hm3_synthetic'` para representar "entry gerada sinteticamente pelo importer .bat" (não chega Discord, não chega user upload — vem direto do HoldemManager3). Cada hand HM3 passa a ter entry_id apontando para a sua entry sintética; backfill retroactivo cria entries para as 1117 mãos antigas.

Plano: 5 peças sequenciais + 1 baseline de matching guard tests para detectar regressões cross-fluxo durante o fix.

### Baseline matching guards (`793b32c`)

11 tests novos em `backend/tests/test_matching_guards.py` antes de tocar em código de produção. Protegem pipeline cross-fluxo durante as 5 peças:

- Guard 1 (6 tests): `_classify_villain_categories` regras A/C/D + excepção #B19 + pré-condição vazia.
- Guard 2 (1 test): `/api/mtt/rematch` processa hand HM3 GG com hashes anónimos (mocka `query()` para isolar phase 2 HH→SS e validar `_enrich_hand_from_orphan_entry`).
- Restantes 4 tests: pipeline `_insert_hand` placeholder collision; `apply_hm3_tag_aliases`; idempotência de re-imports do `.bat`; FRIEND_HEROES regra D.

Os 11 tests passaram em **todas** as corridas pós-cada peça. Zero regressões no matching.

### Peça 1 — idempotência (`398c8a1`)

Schema migration em `main.py` (lifespan): partial unique index `uniq_entries_hm3_synthetic_external` em `entries(external_id) WHERE source='hm3_synthetic' AND external_id IS NOT NULL`. Additive, idempotente, não toca rows existentes.

`entry_service.create_entry` ganha caminho dedicado quando `source='hm3_synthetic' AND external_id IS NOT NULL` — usa `ON CONFLICT (external_id) WHERE source='hm3_synthetic' DO NOTHING` + fallback SELECT para devolver id existente. Caller sempre obtem id válido em re-runs.

### Peça 2 — rematch filtra origin='hm3' (`2ed538a`)

`/api/mtt/rematch` (phase 1: SS → HH) agora filtra `origin='hm3'` quando a entry-fonte é `hm3_synthetic`. Evita que rematches cross-fluxo (SS upload → HH GG anonimizada) tropecem em hands HM3 do mesmo TM.

### Peça 3 — guard DELETE entries 'hm3_synthetic' (`648a789`)

`DELETE /api/entries/:id` ganha guard: bloqueia delete de entries `source='hm3_synthetic'` se houver `hands` ainda linkadas (`entry_id = :id`). Devolve 409 Conflict com `{linked_hands: N}`. Evita orfanizar mãos HM3 por engano.

### Peça 4 — flag has_screenshot_image (`6e5e410`)

`GET /api/entries` ganha campo derivado `has_screenshot_image` (boolean) que é TRUE sse `entry_type='image'` E `raw_json->>'url'` IS NOT NULL. Permite ao frontend distinguir entries-imagem-de-contexto de entries-screenshot-da-mesa sem inferir do `entry_type` (que é igual nos dois casos). Usado em `Inbox.jsx` para o badge "Imagem anexa".

### Peça 5 — importer HM3 cria entry sintética + script backfill (`ce5d173`)

`routers/hm3.py`: o INSERT path do importer agora chama `create_entry(source='hm3_synthetic', entry_type='hand_history', external_id=hand_id, site=parsed['site'], file_name='hm3_bat', status='resolved')` ANTES do INSERT em `hands`. Captura `id` e passa-a no INSERT como `entry_id`. UPDATE path (hand já existe): NÃO toca `entry_id` — preserva o que lá estava (Discord enrichment, backfill etc).

Script `scripts/backfill_hm3_synthetic_entries.py` para hands HM3 antigas com `entry_id` NULL. Dry-run default; `--apply --batch N` executa. Idempotente (UPDATE inclui guard `entry_id IS NULL`; re-runs não duplicam entries).

### Fix retroactivo CHECK constraint (`7ed785f`)

**Descoberto durante o `--apply` em prod.** O 1º batch fails com `CheckViolation` em `entries_source_check`. Peça 1 introduziu o source `'hm3_synthetic'` (+ partial unique index) mas **esqueceu de actualizar o CHECK** em `ensure_entries_schema()`. Implicação dupla:

1. **Backfill bloqueado.**
2. **`.bat` HM3 partido em prod desde o deploy de Peça 5 (`ce5d173`)** — qualquer chamada a `routers/hm3.py:980` `create_entry(source='hm3_synthetic', ...)` aborta a transacção do import.

Diagnóstico via psql contra `ballast.proxy.rlwy.net:37559` (proxy TCP público): a constraint em prod aceitava 10 sources (`'discord', 'discord_bot', 'hm', 'gg_backoffice', 'hh_text', 'summary', 'report', 'manual', 'import', 'screenshot'`) — 3 deles (`discord_bot`, `import`, `screenshot`) nunca chegaram a `schema.sql` mas viviam em prod desde migrations anteriores. Sources em uso em entries: `'discord'` (419), `'hh_text'` (5), `'screenshot'` (1). Nenhuma orfanizada pela nova lista.

Fix: novo statement em `ensure_entries_schema()` no padrão de `routers/hands.py:208` — `ALTER TABLE entries DROP CONSTRAINT IF EXISTS entries_source_check` + `ADD CONSTRAINT entries_source_check CHECK (source IN (...10 valores actuais... + 'hm3_synthetic'))`. Idempotente (`DROP IF EXISTS` + statements independentes com try/except per-statement no loop).

Deploy Railway: 72s até SUCCESS. Constraint verificada via psql pós-deploy.

### Tech debt `#RAILWAY-POSTGRES-PASSWORD-DRIFT` (`450b58a`)

Durante o setup do backfill, diagnosticado que `POSTGRES_PASSWORD` da service Postgres e o password embutido no `DATABASE_URL` da service `poker-app` divergem em 1 caractere (32 vs 31 chars após URL-decode). Credenciais embutidas no URL da poker-app autenticam OK contra o proxy TCP público; credenciais da service Postgres não. Workaround usado: reescrever o URL da poker-app substituindo apenas host/porto pelo proxy público.

Impacto runtime zero (app em prod liga ao DB pelo hostname interno). Impacto operacional alto para ferramentas locais via `railway run` + `DATABASE_PUBLIC_URL`. Severidade 🟡 MED. Só documentado nesta sessão, sem fix.
