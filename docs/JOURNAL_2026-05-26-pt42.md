## Decisões product (validadas pelo Rui antes de codificar)

### Pedido 1 — cortar turn/river

Todas as gerações de zip HRC passam a NÃO modelar betting em turn nem em river. Nessas duas
streets, a única acção possível é check. Implementação no **template canónico** (Opção A da
investigação Q2):

```js
let POSTFLOP_FORCE_CHECKDOWN_AFTER = {
    2: FLOP, 3: FLOP, 4: FLOP, 5: FLOP,
    6: FLOP, 7: FLOP, 8: FLOP, 9: FLOP,
}
```

Pré-pt42 era `{2: RIVER, 3: RIVER, 4: TURN, 5: FLOP}` (5+ vivos já fazia checkdown após flop;
2-3 vivos modelava betting até ao river). Agora todos os live counts forçam checkdown após
flop. `hasNextStreetBetting(ctx)` é o callback que HRC consome e devolve
`street < POSTFLOP_FORCE_CHECKDOWN_AFTER[live]` — i.e., só PREFLOP tem next street com betting,
nunca FLOP/TURN.

### Pedido 2 — regra universal de sizings (recalculada por raise)

**Efectiva por acção (dinâmica):** `min(raiser_remaining, max(active_opponents_remaining)) / BB`.
Substitui a `compute_effective_stack_bb` global (que ficou só para retro-compat). É a
referência única para: gate `eff > 8 BB` do open default; buckets 3-bet 0-25/26-35/35+;
threshold `eff <= 25` do ALLIN-como-2ª-opção.

**Regra aplicada a cada acção do tipo open / 3-bet clássico / squeeze / 4-bet / 5-bet:**

1. **1ª opção** do array = `to_amount_bb` da HH. **Excepção:** se a acção foi all-in
   (`is_all_in = True` via heurística 95% da stack inicial), 1ª opção é `"ALLIN"`.
2. **2ª opção:**
   - Original NÃO ALLIN AND `effective_stack_at_action_bb <= 25` → `"ALLIN"`.
   - Original NÃO ALLIN AND `eff > 25` (ou None) → sem 2ª opção (`[original]` só).
   - Original ALLIN AND non-all-in default existe → o default (por tipo).
   - Original ALLIN AND default None → sem 2ª opção (`["ALLIN"]` só).

**Non-all-in defaults por tipo (só quando original=ALLIN):**

| Tipo | Default | Condição |
|---|---|---|
| **Open** | 2 BB | Só se `eff > 8 BB` E posição ≠ SB ≠ BB (HU "BU/SB" passa) |
| **3-bet clássico** | 2.3/2.7/3.0 × `opener_to_bb` | Bucket por eff: `<26` / `[26,35)` / `>=35` |
| **Squeeze** | 3.0 × `opener_to_bb` | Sempre (se opener_to_bb conhecido) |
| **4-bet** | 2.3 × `previous_raise_to_bb` (= 3-bet anterior) | Sempre |
| **5-bet** | 2.2 × `previous_raise_to_bb` (= 4-bet anterior) | Sempre |

### Implicação: abandonar tabela pt25f

Os helpers `_classic_3bet_band` e `_compute_classic_3bet_overrides` (extensão pt25f, 5
buckets de multiplicador sobre defaults do template, ignorava sizing real da HH) foram
**removidos**. A regra nova traz o sizing real de volta como 1ª opção.

## Implementação

### T1 — Parser: 4 campos novos em cada acção

`backend/app/services/hrc_script_gen.py:_parse_preflop_actions` ganha:

- `previous_raise_to_bb: float | None` — `to_amount_bb` do raise imediatamente anterior na
  sequência. `None` para opens.
- `opener_to_bb: float | None` — `to_amount_bb` do open original. `None` para opens (auto-ref).
- `is_all_in: bool` — `to_amount_chips >= seat_initial_chips × 0.95`. Reusa
  `_ALL_IN_EFFECTIVE_THRESHOLD` de `hrc_node_offset.py` para single source of truth.
- `effective_stack_at_action_bb: float | None` — dinâmica, calculada em
  `_compute_effective_at_action`. Lê `active_players` (set actualizado nos `folds`).

Walker refactorizado: branches `bets` e `raises` partilham `_append_raise_entry` (helper
nested) e fazem state update depois (campos novos lêem state pré-raise).

### T2 — Remoção pt25f

`_CLASSIC_3BET_DEFAULTS`, `_classic_3bet_band`, `_compute_classic_3bet_overrides`,
`_OPEN_ALLIN_THRESHOLD_BB` (constante — mantida, mas semântica nova) removidos da `build_sizings_overrides`.
Branch que aplicava `overrides.update(_compute_classic_3bet_overrides(...))` no fim — fora.

### T3 — Regra universal em `build_sizings_overrides`

5 helpers de default por tipo (`_compute_default_for_open / _classic_3bet / _squeeze / _4bet /
_5bet`) + 2 compositores: `_array_for_raise` (opens/3-bet/squeeze, em BB) e
`_array_for_4bet5bet_in_pot_fraction` (4-bet/5-bet, conversão BB→fração para o template
manter `sizingPot(s)`).

### T4 — Template

Diff cirúrgico de 1 bloco em `mtt_advanced_canonical_2026.js`. `hasNextStreetBetting` e
`getSizingsPostflop` ficam inalterados — o map de `POSTFLOP_FORCE_CHECKDOWN_AFTER` é o ponto
único de controlo.

### T5 — Interacção com `hrc_node_offset.py` (read-only)

`count_lines_for_position` lê `len(overrides[bucket])` — funciona com qualquer dos shapes
novos (`[original]` / `[original, "ALLIN"]` / `["ALLIN"]` / `["ALLIN", default]`). Sem
mudanças necessárias para pt42.

**Tech debt latente descoberto:** `_TEMPLATE_DEFAULT_OPEN_COUNT = 2` em
`hrc_node_offset.py:52`, mas o template default real é `[2]` (1 entrada, pt29 tree-size
control). Quando o gerador não overrida uma posição (ela não fez raise voluntário na HH), o
offset assume 2 linhas, mas o HRC só mostra 1 → `target_node_offset` pode estar **+1 por
posição precedente sem override**. **Não é regressão pt42** (vive desde pt29).
Aberto como `#OPEN-COUNT-DRIFT-HRC-NODE-OFFSET-LATENT` (🟢 LOW; validar empiricamente em
smoke real antes de elevar severidade).
