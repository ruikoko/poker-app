## Arco do dia (a investigação que mudou o desenho)

1. **Premissa inicial (ERRADA):** que o HRC desenha sempre `non_allin + 1`
   (ALLIN implícito sempre) → "521 mãos expostas", e a GG-6083416520 como exemplo.
2. **Correção (lendo o TEMPLATE, não o `_array_for_raise`):** o nº de linhas é
   governado por `getSizingsPreflop`/`shouldAddPreflopAllIn` —
   o ALLIN implícito entra **por stack INDIVIDUAL ≤ limiar**, **não sempre**. A
   GG-6083416520 (todas ≥88 BB > 30) → 1 linha/posição → `count_lines` já estava
   **certo** ali. **O "521" e o exemplo caíram.**
3. **Pergunta do Rui — o limiar é blind-vs-blind?** Confirmado read-only: o gate
   é só `isPreflopFirstVoluntaryRaise` (= é um open) — **geral, qualquer posição
   ≤ 30**, não blind-vs-blind. O comentário pt29 confirma a intenção (filtro
   individual p/ excluir deep de **qualquer** posição).
4. **Verificação anti-regressão (pt29):** a regra nova **25 geral / 30 BvB** NÃO
   ressuscita a explosão do pt29 — esse problema vinha da stack **EFETIVA** (SPR);
   o filtro continua **INDIVIDUAL** (deep 100+ ficam fora: 100>25), e baixar
   30→25 fora dos blinds só **remove** ALLINs (árvore igual ou menor).
5. **Detecção de BvB no template:** possível com a API existente
   (`getActivePlayer`, índices SB/BB, `getNumberOfPlayers`,
   `getPotState().hasPlayerFolded`). Na **tabela de opens** só o open da **SB** é
   BvB → no `count_lines` reduz-se a **SB→30 / resto→25**.

## O fix (`8096f3c`) — `REGRAS_NEGOCIO §19`

- **Template** (`mtt_advanced_canonical_2026.js`): `shouldAddPreflopAllIn` opens
  passa de `30 sempre` → `isBlindVsBlind(ctx) ? 30 : 25`; novo `isBlindVsBlind`.
  Comentário pt29 preservado + nota pt86.
- **`count_lines_for_position`** (`hrc_node_offset.py`): com `stack_bb` espelha o
  template (sizings não-ALLIN abaixo da stack + 1 ALLIN se [explícito ∨ stack ≤
  limiar ∨ size ≥ stack colapso] + Complete-SB; limiar SB→30/resto→25).
  `stack_bb=None` = **legacy intacto** (36 testes antigos verdes). Novo
  `derive_position_stacks_bb` (= `getChipsRemaining/BB`); `compute_target_node_offset`
  ganha `position_stacks_bb`; `build_queue_zip` deriva-as e passa.
- **+12 testes** (mapa completo + a mão real). **48 passed** no ficheiro; suite
  restante 330 passed (3 falhas pré-existentes Postgres local).
