## Lições

1. **Race condition Vision-vs-trigger é real e reproduzível.** `asyncio.sleep(10)` não basta com batches >20 entries. Workaround: re-sincronizar após Vision terminar. Solução robusta: aguardar todas as Vision tasks ou trigger periódico.

2. **Tech Debt #3 expõe padrão arquitectural problemático.** Cada path de criação de hand novo precisa duplicar boilerplate de trigger Bucket 1 (`discord.py`, `hm3.py`, **falta `import_.py`**). Solução robusta: centralizar em `hand_service.py` (saltado por hot path no Bucket 1).

3. **Estimativas de tempo recalibradas.** Trabalho frontend/backend pequeno com modelo já feito = ~5min, não 30-45min. Memória registada.

4. **Validação visual UI vs validação BD são complementares mas separadas.** BD diz "anexado", UI diz "renderizado". Bug visual `img_b64=NULL` (entry 31) só apanhado por validação visual real, não por queries.

5. **Snowflake Discord ≠ unix_ms.** Cálculo manual mental falhou na 1ª tentativa (off-by-7-days). Lição: usar Python explícito (`datetime + bitshift`) e decodificar back para confirmar antes de aplicar UPDATE.

6. **Estado em movimento durante audit gera leituras inconsistentes.** Quando a contagem de `hands` divergiu entre Q1 e Q6 (15057 vs 15530), confirmou que pipeline ainda processava ZIP em background. Lição: antes de audit final, verificar `entries WHERE status='new'` e estabilidade da contagem `hands` via 2 leituras consecutivas.
