#### Tech Debt #2 — `img_b64=NULL` Gyazo (commit `9baec67`)

**Investigação**: 4 entries Gyazo image (31/38/40/66) — 3 com `img_b64` válido (~890KB-959KB), 1 (entry 31) com `mime=NULL` e `img_b64=NULL`. Único factor distintivo evidente: o ID Gyazo da entry 31.

**Teste real Gyazo via Python (3 hipóteses A/B/C)**:

| Entry | `i.gyazo.com/<id>.png` | `i.gyazo.com/<id>.jpg` |
|---|---|---|
| 31 (FAIL) | 404 `"Gyazo image is not found"` | **200 image/jpeg 2.2MB** |
| 38, 40, 66 (OK) | 200 image/png ~700KB | 404 |

**Causa raiz: Hipótese B confirmada**. `attachments.py:93` constrói **sempre** `f"https://i.gyazo.com/{gid}.png"` — assume PNG. **Fix**: loop sobre extensões `.png` → `.jpg` → `.gif`. Se nenhum HEAD acertar, `return None` directamente em vez de cair para URL HTML — elimina silent fallback (causa-mãe). Log warning descritivo (gid + URL original) para diagnóstico futuro.

**Validação**: deploy live `13:32:36 UTC`, post-backfill: `att_id=8 mime='image/jpeg', b64_len=2970828`. Estado global: `4/4 com_b64`, `0/4 sem_mime` ✓.

**Parte C — endpoint `/api/villains/categorized` (commit `c409677`)**:

Query CTE com `COUNT(*) OVER()` inline (1 query, sem 2ª COUNT separada):
```sql
SELECT nick, sd_count, nota_count, friend_count, total_count,
       last_seen, sites, dates, COUNT(*) OVER() AS total_villains
FROM (
  SELECT hv.player_name AS nick,
         COUNT(*) FILTER (WHERE hv.category = 'sd')     AS sd_count,
         COUNT(*) FILTER (WHERE hv.category = 'nota')   AS nota_count,
         COUNT(*) FILTER (WHERE hv.category = 'friend') AS friend_count,
         COUNT(*)                                       AS total_count,
         MAX(h.played_at) AS last_seen,
         ARRAY_AGG(DISTINCT h.site) AS sites,
         ARRAY_AGG(DISTINCT h.played_at::date ORDER BY h.played_at::date DESC) AS dates
  FROM hand_villains hv JOIN hands h ON h.id = hv.hand_db_id
  WHERE hv.hand_db_id IS NOT NULL
    AND h.played_at >= '2026-01-01'
    AND (%(search)s IS NULL OR hv.player_name ILIKE %(search_like)s)
  GROUP BY hv.player_name
  HAVING (CASE %(category)s
            WHEN 'sd'     THEN COUNT(*) FILTER (WHERE category='sd')     > 0
            WHEN 'nota'   THEN COUNT(*) FILTER (WHERE category='nota')   > 0
            WHEN 'friend' THEN COUNT(*) FILTER (WHERE category='friend') > 0
            ELSE TRUE END)
) AS v
ORDER BY total_count DESC, last_seen DESC NULLS LAST
LIMIT %(limit)s OFFSET %(offset)s
```

Query params: `?category=all|sd|nota|friend` (default `'all'`, validado via pattern regex), `?search=`, `?page=&page_size=`. Posteriormente adicionado `?site=` (ver Parte D).

## Decisões arquitecturais importantes

### Tech Debt #6 — Opção D vs A/B/C

Análise de trade-offs UX vs robustez:
- **A** (sleep 90s): bloqueia request HTTP do Rui 90s — UX-degrading no caminho mais comum.
- **B** (await all Vision): refactor `process_replayer_links` complexo, ratio risco/ganho mau.
- **C** (cron periódico): complexidade desproporcionada, recursos sempre consumidos.
- **D** ✓: 5min código, low-risk, cobre 99% dos casos, request HTTP retorna imediatamente.

### Tech Debt #5 — Backend pré-resolve (B) vs Frontend refactor (A)

Quando algoritmo "deveria funcionar" mas não funciona em produção, **mover responsabilidade para backend** elimina dependência de cache/build state e dá garantia determinística. Frontend mantém fallback `raw_resolved || raw` graceful.

### Tech Debt #4 — Schema com coluna `category` vs derivação on-the-fly

Alternativa rejeitada: derivar categorias via JOIN dinâmico com `hands.hm3_tags`/`discord_tags`/etc. Argumentos contra:
- Performance pior (cada query Vilões teria que avaliar regras A∨B∨C∨D em runtime).
- Mais complexidade de SQL nas queries.
- Sem rasto auditável de "porquê este villain entrou nesta categoria" no momento do INSERT.

Decisão: persistir categoria no schema (via INSERT correcto pelo helper centralizado). Backfill defensivo recategoriza histórico existente.

### Tech Debt #4 — 4 tabs (não 3) + filtro site mantido

Spec inicial era 3 tabs (SD/Notas/Amigos). Rui acrescentou tab `Todos` durante a validação Q6 — útil para visão geral sem filtro de categoria. Filtro site (`siteFilter`) também mantido (Q3) — Rui confirmou usar para isolar GGPoker vs outras salas.

## Lições

1. **Race condition Vision-vs-trigger é resolúvel sem refactor delicado.** Opção D (2x trigger fire-and-forget) é mais robusta que aumentar sleep ou refactorizar awaits — não bloqueia UX, idempotência garante zero side-effects.

2. **Silent fallback é pior que erro explícito.** O bug `img_b64=NULL` Gyazo só se revelou porque o helper caía silenciosamente para HTML em vez de retornar None. Eliminar o fallback não só corrigiu, como fez o código mais debuggável.

3. **Backend pré-resolve > frontend refactor cego.** Quando algoritmo "deveria funcionar" mas não funciona em prod (cache busting já descartado), parar de adivinhar e mover para backend determinístico. Frontend pode ainda falhar mas a fonte de verdade é única.

4. **Spec rigorosa antes de código poupa retrabalho.** Tech Debt #4 teve 3 rondas de perguntas antes de qualquer linha de código (regras canónicas, coexistência multi-cat, hero exclusion semantics, paginação, search, sort, cores, tabs). 0 linhas re-escritas, 4 commits limpos.

5. **`ON CONFLICT` em indexes parciais precisa de `WHERE`.** `ON CONFLICT (cols)` falha com "no unique or exclusion constraint matching" se o constraint é parcial.
