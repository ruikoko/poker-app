## 2. 🚨 Achado crítico 1 — regressão do anchor (latente)

Investigação do código + simulação na BD prod:

- `tournament_meta.upsert_tournament_meta` é **GG-only**, `start_time = MIN(played_at)`, gatilho único `import_.py` (HH GG zip). Não populado por HM3/backoffice/lobby. Único consumidor funcional de `meta.start_time` = resolver TIER 1 (`GET /meta` expõe-no mas o frontend só lê `starting_stack`).
- **Gap quantificado (99 torneios GG):** `lag = meta.start − TS.start` é **sempre ≥ 0** (57/99 >1min); mediana ~7min, p90 ~42min, max 3.2h. Variável (não offset constante) → confirma a re-diagnose pt39 (não-TZ, é `MIN(played_at)` vs arranque).
- **★ Simulação do novo TIER 0 (pt39 `35286c1`) nos 3 lobby `tm_not_found` pt37:**
  - Deepstack $125 (post 16:07, TS start 15:05) → **resolve certo** (post depois do start).
  - Daily Hyper $80 18-Mai (post 16:15, TS start 16:45) → **0 rows / `tm_not_found`** (instance certo ainda não começou; anterior >24h).
  - Daily Hyper $80 19-Mai (post 16:17, TS start 16:45) → **mis-resolve para 18-Mai** (`ORDER BY start DESC LIMIT 1` agarra o instance que já começou) → **`tournament_payouts` errado**.
- **Causa-raiz:** o anchor `start_time ≤ posted_at` assume **SS tirada durante o jogo** (verdade p/ **table-ss**, `captured_at` após o start). Mas os **lobby SS são tirados na fase de inscrição** → o torneio começa **~30min depois** do post. Logo o instance certo é excluído e o do dia anterior é escolhido.
- **Latente:** `M1=0` — nenhum lobby `success` processado pós-deploy (a última actividade de lobby foi 23-Mai, pré-deploy). **Mas blast radius ~24** `tm_not_found` acumulados (17 de 23-Mai + 7 de 19-Mai) disparariam na próxima actividade.

## 5. 🚨 Achado crítico 2 — bounties errados no converter GG→PS

Descoberto pelo Rui ao testar o Track B (Big Game $215 → Hero `€250` no HH). Investigação read-only:

- `_HERO_BOUNTY_DEFAULT_USD = 250.0` (`queue_export.py:488`, = bounty base do **$525** Big Game) aplicado a **todos** os torneios. `_inject_bounties_ps_format` (~521-563): cada Seat GG leva `, €<X> bounty)`, Hero = `max(Vision, 250)`.
- **Sintoma:** Big Game **$215** tem `buy_in_bounty=$100 USD` no TS, mas o Hero sai `€250` — **moeda E magnitude erradas**.
- **Pior que o documentado:** (a) **sem gate de formato** → vanilla GG (`Daily Hyper $80`, `buy_in_bounty=None`) também recebe **€250 fantasma**; (b) **vilões sempre a €0** (Vision não lê HHs GG anonimizados); (c) afecta **batch (`/api/queue/hrc`, watcher) E per-mão** — **NÃO é regressão pt40**, vive desde FASE-1/pt28; o endpoint per-mão herdou.
- **Moeda `€` é intencional** (workaround do parser HRC, validado pelo Rui — rejeita `$`).
- **Fix desbloqueado:** o TS já tem `buy_in_bounty` por `tournament_number` ($215→100, $108→50, $44→20). Debt `#HERO-BOUNTY-FROM-TS-DERIVATION` subido a **🔴 HIGH 🚨** — **foco pt41** (afecta cada mão GG estudada hoje).
