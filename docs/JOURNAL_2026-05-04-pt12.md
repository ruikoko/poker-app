## #B33 — investigação e fix

### Sintoma
Pt11 Onda 8 (delete + re-import GG ZIP) revelou regressão: 1/7
hands canal nota (`GG-5891409707`, id=2297) ficou sem `discord_tags`
nem villains, apesar da entry Discord (id=30) existir. Diferença
vs pt10 (que tinha 7/7 OK).

### Investigação read-only
3 hipóteses iniciais (auto-rematch order, `replayer_link` vs
`hand_history`, caminho de inserção) — todas refutadas por evidência
empírica.

Hipótese vencedora (4ª): regex em `screenshot.py:307` exigia prefixo
`TM` literal (`r'TM(\d+)'`). Vision às vezes devolve `TM: 5891409707`
(só número) em vez de `TM: TM5891409707` (com prefixo). Quando
prefixo ausente, regex falha → `result["tm"]=None` →
`backfill_ggdiscord` (`discord.py:825-827`) skipa silenciosamente.

Magnitude: 2/26 entries Discord da pt11 afectadas (id=30, id=36).
Padrão: ~7-8% das `replayer_link` têm omissão do prefixo.

### Fix (commit `e7d88b2`)
1 linha em `screenshot.py:307`:
- Antes: `r'TM(\d+)'`
- Depois: `r'\b(\d{8,12})\b'`

Word-boundaries protegem contra timestamps acidentais (>12 dígitos).
Aceita 4 formatos do Vision: `TM12345`, `12345`, `TM 12345`, `TM: 12345`.

### Backfill retroactivo
1. UPDATE `entries` SET `raw_json.tm` para id=30 e id=36.
2. `append_discord_channel_to_hand` → populou `discord_tags`.
3. `_enrich_hand_from_orphan_entry` → `mm='anchors_stack_v2'` +
   `anon_map` populado (8 entries cada).
4. `apply_villain_rules` disparou automaticamente:
   - hand 2297 (canal `nota`): 2 villains criados (`MeetU@TheAf..`,
     `Nighting@le_`) via Regra C.
   - hand 2083 (canal `icm-pko`): 0 villains (correcto — A∨C∨D não
     dispara em canal não-nota).

## Decisão de produto: vilão principal

Durante a sessão, Rui levantou tema relacionado: regra para
identificar "vilão principal" quando hand tem múltiplos villains
(estrelinha ⭐ na UI página `/villains`).

Spec definida (pendente implementação):
- Vilão principal = aquele que chegou mais longe na mão.
- Hierarquia: showdown > river > turn > flop > preflop.
- Sem tie-break: se múltiplos chegaram à mesma street, todos são
  principais (estrelas múltiplas).

Adiado para sessão dedicada (~1-2h: backend + frontend + DB migration).

## Vilão Principal — implementado

### Decisão final
Decisão B (filtrar candidates, sem flag `is_primary`, sem migration).

`apply_villain_rules` em `villain_rules.py` passa a filtrar candidates
por street máxima atingida. Hierarquia:
- 5 = showdown (river + cards reveladas, NÃO cards mucked GG)
- 4 = river
- 3 = turn
- 2 = flop
- 1 = preflop
- 0 = sem dados (apa placeholder — todos passam, edge case)

Tie-break: SEM. Múltiplos com mesma street máxima → todos criados.

### Implementação (commit `0ebacfd`)
- Adicionada `_street_reached(actions, cards)` em `villain_rules.py`.
- Adicionada `_filter_to_furthest_street(candidates, apa)`.

## Stack Inicial GG canónico (commits `68a9e8a` + `799864e` + `457048f` + `a2158c3`)

### Bug observado
Página `/estudo` modo "Por Tags" mostrava SI errado:
- "Bounty Hunters Deepstack Turbo $88": SI 540,631 (real: 20,000).
- "Bounty Hunters HR $525": SI 18,046 (real: 27,500).

### Causa raiz
Frontend calculava SI como `apa[hero].stack` da 1ª hand cronológica
DO BUCKET FILTRADO (tagged hands only). Para hands GG via ZIP HH
completo, isso era OK. Mas a vista filtra por tags, logo 1ª hand
do bucket != 1ª hand real do TM.

### Decisão de produto
Persistir metadata por torneio em tabela dedicada `tournaments_meta`
(separada da tabela `tournaments` legacy de P&L summary).

### Schema `tournaments_meta`
- `tournament_number` (PK component)
- `site` (PK component, restrito a GG)
- `tournament_name`, `buy_in`, `currency`
- `tournament_format` (Vanilla/PKO/Mystery KO/Super KO)
- `starting_stack`
- `start_time`, `hand_count`, `updated_at`

### Restrição GG-only
HM3 `.bat` só importa hands com tag → 1ª hand cronológica em BD
nunca é a real do torneio para Winamax/PS/WPN. Logo, SI só faz
sentido para GG (ZIP HH completo). Guard explícito em
`upsert_tournament_meta`.

### Implementação
1. Migration: `ensure_tournaments_meta_schema()` em `main.py:lifespan`.
2. Função canónica `services/tournament_meta.py:upsert_tournament_meta`
   com detecção de currency (símbolo $/€ no nome > default per site).
3. Hook em `_run_zip_import`: agrupa hands inseridas por TM, chama
   upsert para cada.
4. Endpoint `GET /api/tournaments/meta?tms=A,B,C`.
5. Frontend `Hands.jsx`: lookup com fallback graceful ao cálculo
   client-side se cache miss.
6. Backfill 26 TMs → 20 rows GG (após filtro): 100% populadas.

### Validação
- Para TMs 280293083 e 280317122 (ambos "Bounty Hunters Deepstack
  Turbo $88"): `starting_stack=20000` ✓.
- Validação visual em prod confirmada pelo Rui.
