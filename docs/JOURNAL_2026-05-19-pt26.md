### Extensão `swap_and_smoke.py` (sem commit — `_local_only/`)

`swap_and_smoke.py` estava em estado pt23 (4 SWAPS only). Pt25e/pt25f
introduziram 13 funções novas + 15 constantes que não existem no `.pyc`
Baltazar e portanto **não podem ser swapped** — têm de ser bound em
module globals em runtime.

**Estratégia escolhida: trampoline.** Em vez de bytecode patching para
inserir `STORE_NAME` no body do mod_code (complexo, line-table delicado,
opcode-version-dependent), construí um trampoline que é o novo módulo
do `.pyc`:

```python
import marshal as __m
_O = b'<marshal.dumps(swapped_baltazar_mod_code)>'   # ~50 KB
_P = b'<marshal.dumps(patched_funcs_mod_code)>'      # ~30 KB
exec(__m.loads(_O), globals())   # 1) Baltazar body — helpers, constantes
exec(__m.loads(_P), globals())   # 2) patched_funcs body — new funcs + consts
del __m, _O, _P
```

Os dois `exec()` partilham o mesmo `globals()` dict do trampoline.
Resultado idêntico a bytecode-merge mas zero opcode-juggling. Launcher
Apr19 (`hrc_watcher_apr19_launcher.pyc`) continua a fazer
`marshal.load(f)` directo (header-less), sem mudanças necessárias.

A swap pt23 dos 4 code objects em `mod_code.co_consts` está
**preservada** como defence-in-depth: se o segundo `exec()` falhar, o
primeiro já correu com as 4 fixes.

**Diff exacto** (output do script):

## Re-classificação do problema do equity_model

Após análise mais cuidada da queixa original do Rui (entregue 19 Maio
noite), o bug reportado **não é FT vs MTT mismatch**. É outro:

**Sintoma real:** mão
`WN-4690815078549684227-208-1778279040` é 13-left num torneio Winamax
6-max. Backend escreveu `meta.json` com `players_left: null` (por o
lobby_processing_log não ter capturado a SS do lobby — Vision API
falhou). Watcher caiu no workaround `#HRC-MTT-OTHER-TABLES-INFO`
aberto em pt23: detectou `equity_model='multi_table_icm'` sem
`players_left` e saltou a página MTT Stacks com Other Tables = 0. HRC
tratou o torneio como tendo apenas 4 jogadores totais (= jogadores
sentados na mesa do hand). ICM cálculo inválido — strategies dessa run
não são confiáveis para esta mão específica.

**Cadeia causal:**
1. Vision API do lobby falhou em processar o SS do torneio para esta mão
   (não chegou a popular `lobby_processing_log`, ou populou com erro).
2. `_resolve_players_left(hand, payout_blob)` em
   `queue_export.py` devolveu `None` (lookup falhou).
3. `meta.json` saiu com `players_left: null`.
4. Watcher seguiu o branch `elif equity_model == 'multi_table_icm'` (pt23
   Bug E fix em `patched_funcs.py:setup_hand`).
5. `click_rel(wpos, *BTN_NEXT)` skip da página MTT Stacks deixou Other
   Tables em 0 (default HRC).
6. HRC considerou: 4 jogadores na mesa = 4 jogadores no torneio
   total. Equity model `Multi Table ICM` com 0 mesas paralelas →
   matematicamente equivalente a FT ICM de 4 jogadores.
7. Cálculo distorcido vs realidade (13 jogadores em ~2-3 mesas).

**Insight chave:** o bug é **independente** da escolha
FT/MTT do equity_model. Mesmo se o backend tivesse classificado como
FT (via tag), o HRC ainda precisaria de saber que há 13 jogadores no
torneio para calcular ICM correctamente. O default Other Tables = 0 é
o erro real.

**Gatekeeper:** `#HRC-MTT-OTHER-TABLES-INFO` (aberto em pt23 Bloco 7,
aceite como compromisso na altura). Workaround actual sub-óptimo
continua activo até **2 fixes paralelos** estarem prontos:

- **Vision API confiável** — capturar `players_left` consistentemente
  do lobby SS (ver `#VISION-LOBBY-API-FAILURE` abaixo).
- **Watcher escreve Other Tables não-zero** — quando `players_left`
  está disponível no meta.json, calcular `other_tables_count = ceil((players_left
  - max_players) / max_players)` e escrever no campo. Source-side fix
  em `handle_mtt_stacks_page` ou função paralela. Não trivial — coords
  do campo e Generate button precisam calibração smoke.

## Erro do Web auto-registado

Web (Claude Web) interpretou mal a queixa original do Rui sobre
"equity_model errado" como FT/MTT mismatch, levando a Code a investigar
em direcção tangencial (proposta de 4 opções FT-auto-classification que
não resolveriam o problema real).

**Causa:** o vocabulário "ICM Malmuth" vs "Multi Table" no painel HRC
foi confundido como classificação FT/MTT canónica do backend. Na
realidade, o ponto do Rui era a **contagem de jogadores no torneio
visto pelo HRC** — distorcida porque `players_left` faltava.

**Lições aprendidas (Web):**

1. Resistir à tentação de "reconhecer padrões" sobre o que o user está
   a queixar-se antes de confirmar entendimento literal. Em qualquer
   queixa sobre "config errada no HRC", confirmar primeiro O QUE o HRC
   viu (pages do wizard, valores escritos, screenshot do estado real)
   antes de assumir qual é o bug.
2. Vocabulário poker do Rui é técnico e preciso — "ICM Malmuth" no
   painel HRC pode ser usado como sinónimo de "modelo correcto para
   FT" mas isso é estado **interno do HRC**, não da BD. Backend
   `equity_model = "malmuth_harville_icm"` é uma string-tag, não uma
   garantia de classificação correcta do contexto multi-table.
3. Quando Code devolve análise extensa que aponta para "design
   intencional, não regressão", confirmar com o Rui que o problema foi
   correctamente entendido **antes** de pedir opções A/B/C/D. Custo
   desta vez: uma ida-e-volta perdida com Code numa direcção que não
   resolve nada.

**Confirmação do design tag-based como CORRECTO e não relacionado com
este problema:** a investigação FT/MTT (descartada) continua válida
como contexto histórico. O design tag-based desde pt23 (`c3cc66b`),
extensão pt25f (`92778bd`), é canónico e não foi tocado. Não há fix a
fazer aí.
