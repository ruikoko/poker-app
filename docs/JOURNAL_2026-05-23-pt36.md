## Origem — investigação do pipeline HRC ponta-a-ponta (read-only)

A sessão começou com um pedido do Web para mapear o pipeline HRC completo, do
sync Discord até ao fim da 2ª run (secções A-K). Achados transversais:

- **Correcção factual:** a Vision que lê o **link de replayer GG no Discord**
  é **OpenAI `gpt-4.1-mini`** (`backend/app/routers/screenshot.py`, via
  `_run_vision_for_entry`), **não** Claude Sonnet. Sonnet é a Vision do
  pipeline **#lobbys** (`services/lobby_vision.py`) e do **backoffice**
  (`services/tournament_result_vision.py`), que produzem `tournament_payouts`.
- O link Discord nunca preenche `hands.raw`: extrai og:image + Vision + match;
  o `raw` vem do import manual de HH (`origin='hh_import'`), cosido por
  `hand_id` partilhado via `_insert_hand` (apaga placeholder GGDiscord).
- A queue HRC (`GET /api/queue/hrc`) selecciona por tags + study_state + site
  + janela; sem flag explícita. HH ausente → skip `no_raw_hh` silencioso em
  `build_queue_zip`.

## Achado disparador

Na secção K (2ª run), confirmou-se que o watcher só dispara a 2ª run
(Selected Subtree) **se `aggressor_real_action != None`**
(`tools/watcher_src/patched_funcs.py:1987`). Mãos sem raiser preflop
(limp pots, walk-to-BB) ou com aggressor inutilizável davam `None` →
**exportavam só 1 run** → qualidade inconsistente da biblioteca GTO.

## Decisão do Rui

**Sempre 2 runs** (Opção A de produto). Pré-requisito da Fase 2 do GTO Brain:
a biblioteca tem de ser uniforme antes de se ligar o auto-import de trees.

## Investigação técnica

- `aggressor_real_action` é um **gate booleano** no robot — só a presença
  importa; os campos `position`/`size_bb`/`type` **não são lidos** no flow da
  2ª run (serviram para o backend calcular `target_node_offset`).
- `target_node_offset` (em `meta.json`) **deriva** de `aggressor_real_action`
  via `compute_target_node_offset`; é `None` quando o aggressor é `None`, ou
  quando a position é inutilizável (`None`/`"BB"`/fora de
  `strategy_table_positions`).

## Opções de fallback ponderadas

- **D1 — sentinela no backend:** quando não há aggressor utilizável, escrever
  um `aggressor_real_action` sentinela a apontar à **raiz** da Strategy Table
  (`positions[0]`, offset 0). Fix só em `queue_export.py`; **sem recompilar o
  `.exe`**.
- **D2 — desacoplar no robot:** mudar o gate em `patched_funcs.py` para
  dispatch incondicional. Exigiria recompilação do `.exe`.
- **Escolha: D1** (menor risco operacional, não toca no robot/Beelink).

## Esboço final — 4 zonas em `queue_export.py:build_queue_zip`

1. **Zona 1** — derivar `seats_at_table`/`positions` cedo + **skip
   `no_seats_at_table`** (HH sem button / <2 seats = malformada → não vai ao
   robot) antes de escrever `hh.txt`.
2. **Zona 2** — `aggressor_real_action` com fallback unificado: `real` /
   `fallback_root` / `fallback_unusable_position`; sentinela
   `{type:"fallback_root", position:positions[0], size_bb:None, source:<tipo>}`
   nos fallbacks; o caso `real` preserva a estrutura legacy (sem `source`).
3. **Zona 3** — `target_node_offset` computado **sempre** (raiz=0 no fallback).
4. **Zona 4** — campo novo `aggressor_source` no `manifest.hands_included[*]`.

## 2 ajustes do Web ao esboço

- **(a)** Recuo inicial sobre skip vs fallback degradado: o Web confirmou o
  **skip** `no_seats_at_table` — HH malformada não deve ir ao robot só para

- **(b)** O `source` interno do dict sentinela passa a **reflectir** o tipo
  (`fallback_root` vs `fallback_unusable_position`), em vez de um literal fixo;
  o caso `real` fica **sem** `source` (estrutura legacy intacta).
