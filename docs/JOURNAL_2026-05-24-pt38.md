## 1. Investigação read-only — flow do `players_left` até ao HRC

Mapa do estado actual (zero código): como o `players_left` é capturado,
resolvido por mão e consumido pelo watcher.

- **Fonte única real:** `lobby_processing_log.players_left`, escrito **só** pelo
  pipeline de SS de lobby (`#lobbys`, Vision Anthropic `claude-sonnet-4-6`).
  Não existe em `hands` nem `tournament_payouts`.
- **Resolução por mão:** `queue_export._resolve_players_left` (on-the-fly no
  export) → lookup em `lobby_processing_log` por `tournament_number`, **sem
  janela temporal** (LIMIT 1 pela leitura mais recente). Uma SS de lobby aplica
  o mesmo número a **todas** as mãos do torneio.
- **Consumo watcher:** `meta.json.players_left` → força `stage=MTT` → escreve em
  "Remaining Players" (`handle_mtt_stacks_page`). **Other Tables nunca era
  escrito** → cadeia `#HRC-MTT-OTHER-TABLES-INFO`.
- **Qualidade:** fonte escassa + ~34% falha Vision (`#VISION-LOBBY-API-FAILURE`,
  pt27). Maioria das mãos GG elegíveis sem `players_left`.

Conclusão: para mover a agulha era preciso uma fonte de `players_left`
**por mão** e **alinhada temporalmente** — daí o pipeline de SS de mesa.

## 2. Descoberta empírica — `#HRC-MTT-OTHER-TABLES-INFO` é falso positivo (`6d7a2e0`)

Rui mostrou 2 SS do HRC (antes/depois de OK no sub-popup "Generate MTT Stacks"):
- **Antes:** Other Tables Players = 0.
- **Depois (Remaining Players=313):** Other Tables Players = **305** (313−8),
  Other Tables Chips = **54 762 354** (auto-calculado).

→ **O HRC auto-calcula "Other Tables" a partir de Remaining Players.** O watcher
não precisa de escrever esse campo. Re-classificado:
- `#HRC-MTT-OTHER-TABLES-INFO` → ✅ **FALSO POSITIVO** (cadeia pt26 errada a
  partir do 3º elo). Entrada histórica pt23 preservada.
- Novo `#HRC-MTT-STACKS-PAGE-SKIPPED-ON-NULL-PLAYERS-LEFT` (🔴 HIGH): quando
  `players_left=None` o watcher salta a página MTT-Stacks → defaults → ICM
  colapsa a FT dos sentados. Causa raiz = falta de `players_left` por mão.

## 4. Design do pipeline (esboço aprovado, 5 decisões)

Multi-site, 2 mecanismos (manual UI + automático `.bat-like`), endpoint backend
partilhado. Decisões aprovadas pelo Web:
1. Match **temporal mão→resolver-desambigua** (temporal primeiro; resolver só
   para multi-tabling).
2. TZ de captura **Europe/Lisbon** (configurável `TABLE_SS_CAPTURE_TZ`).
3. `hands.context_table_ss_id` **INTEGER soft-FK**.
4. Auth **cookie** (partilhada pelos 2 mecanismos).
5. Janela de match **±5 min** (`TABLE_SS_MATCH_WINDOW_S = 300`).

Faseamento: A (backend) → B (UI) → C (cliente automático).

## 9. Diagnóstico do mapeamento + fix + reset (`0d0ec30` + reset BD)

Investigação do `vision_json` das 7 rows revelou o **bug de prompt** (não de
mapping — a app guarda verbatim):

**Formato real dos painéis (confirmado pelo Rui):**
- **Winamax:** `Rank: <hero_rank> / <players_left> (<itm_places>)` — 3 números.
- **GGPoker:** `My Rank: <hero_rank> / <players_left>` — 2 números (sem ITM no painel).

O prompt v1 ensinava a barra como *left/entrants* (exemplo `Players: 71/124`) e
**não tinha `hero_rank`**. Resultado: a Vision foi **inconsistente** entre SSs —
`players_left` guardado ficou **errado em 3 de 4 success** (ODYSSEY 71=rank;
GALACTICA 7=itm/parênteses; GG Bounty Hunters 813=rank); HIGHROLLER acertou por
coincidência (rank=left=8). **Bloqueava o HRC** (o `_resolve_players_left`
recebia `players_left` errado).

**Fix (`0d0ec30`):**
- Novo campo `hero_rank` + regra reescrita com **mapeamento explícito por site**;
  `players_left` = SEMPRE depois da barra; `total_entries` só se contador
  separado/explícito; exemplo enganador removido. `hero_rank` só no `vision_json`
  (sem coluna). +3 testes. Suite 618 → **621 PASSED**.
- **Reset BD** (transacção única, COUNT antes): `UPDATE hands SET
  context_table_ss_id=NULL` (3 rows) + `DELETE FROM table_ss_processing_log`
  (7 rows) → 0 + 0. Dados com o mapeamento errado limpos; SSs re-uploadáveis.

## Notas factuais

- **Painéis:** Winamax `Rank: rank / players_left (itm)`; GG `My Rank: rank / players_left`.
- **Plano de dados distinto:** o pipeline SS de mesa alimenta **só** o
  `players_left` por mão (`table_ss_processing_log` → `hands.context_table_ss_id`).
  **NÃO** escreve em `tournament_payouts`.
- **TS GG vs TS Winamax (comparação campo-a-campo, pt38):** o **parser TS
  implementado na app é GG-only** (cobre os Tournament Summaries da GG). Mas o
  **TS Winamax existe** como ficheiro com info parcialmente comparável — Rui
  mandou exemplo `BATTLE ROYALE_1098231707` e comparámos:
  - **TS Winamax** tem: buy-in breakdown, total entries, prize pool, start_time,
    posição final do hero, estrutura **completa** de blinds, `bountyType`
    explícito (`Type: knockout`). **Falta:** `hero_prize` amount (Rui ficou
    fora ITM) e estrutura de prémios por posição.
  - **TS GG** tem: `hero_prize_amount`. **Falta:** estrutura completa de blinds
    e `bountyType` explícito.
  - **Conclusão prática:** **nenhum** dos dois TS substitui o lobby SS para
    popular `tournament_payouts` (estrutura completa de prémios por posição),
    mas ambos seriam viáveis para **identificação + dados macro** se parseados.
    **Parse TS Winamax = follow-up potencial.**
  Logo a SS de mesa **não substitui** a lobby SS para `tournament_payouts`
  (payout structure ≠ players_left context — planos distintos).
- **Colisão observada:** INTERSTELLAR + EXPLORER ligaram à **mesma** mão
  (`WN-…39…`) e o resolver devolveu o **mesmo** `tournament_number` (`1100185162`)
  para 2 nomes diferentes → `#TABLE-SS-RESOLVER-COLLISION`.
