## Investigação dos 3 `tm_not_found` vanilla GG

- **Autenticação Railway interactiva:** `railway login` num shell separado do Rui; a config fica por-utilizador e o terminal do Code herda-a (comandos `railway` não-interactivos passam a funcionar).
- **Script SQL diagnóstico read-only** contra a BD de produção (psycopg2 + proxy público `ballast.proxy.rlwy.net`; password viva vem do `DATABASE_URL` do serviço `poker-app` — o `DATABASE_PUBLIC_URL` do serviço Postgres tinha password **stale**).
- Análise caso a caso de `1505967173032607784`, `1506327372629282879`, `1506329968781557890` (todos GGPoker, todos `start_time_iso=None`).
- **Causa raiz dupla:**
  1. **TIER 0 — igualdade estrita** em `prize_pool`/`total_players`. A Vision lê o lobby **em tempo real** (garantia anunciada redonda + inscritos parciais a meio da late-reg) vs `tournament_summaries` **pós-jogo** (pool real nunca-redondo + total final). Ex.: `1505967173032607784` Vision pool=5000/players=46 vs TS mesmo-dia `tn 284939855` pool=7580.80/players=103. Nunca pode bater — mas o torneio **existe** (TIER 0 só-nome devolve ≥2 rows).
  2. **TIER 1/2 — janela inutilizável** sem `start_time_iso`. A Vision não lê a hora de início (consistente); o fallback usa `[posted_at−12h, posted_at−30min]`, que **exclui o candidato** (o torneio começa *depois* do post — o user posta a SS durante o jogo). Candidatos do mesmo dia começam às 16:45–17:16; a janela termina às 15:37–15:47.
- **Achado adicional:** `tournament_summaries.start_time` vs `tournaments_meta.start_time` para o **mesmo** `tournament_number` diferem ~2h (`tn 284491487`: 15:05 vs 17:03). **Bug de timezone** num dos parsers — bloqueia matching temporal seguro.
- **3 tech debts HIGH abertos:** `#RESOLVER-TIER0-STRICT-EQUALITY`, `#RESOLVER-TIER12-WINDOW-NO-START`, `#START-TIME-TIMEZONE-INCONSISTENCY`.
