**Última actualização:** 26 Abril 2026
**Âmbito:** Validação ponta-a-ponta dos 4 pipelines de ingest: SS upload manual (SSMatch), Discord, HM3 .bat, e anexos imagem ↔ mão (Bucket 1).

Este guia é o **playbook** para validar que cada pipeline está a funcionar correctamente em produção. Cada secção tem:
1. Descrição do pipeline
2. Pré-condições (dados de entrada)

4. Validação por Fase com queries SQL específicas

## Princípios gerais

### Quando validar
- Após **qualquer fix arquitectural** num dos 3 pipelines
- Após **import grande** que envolva múltiplas origens (ZIP HH + SSs simultaneamente)
- **Periodicamente** (mensal) para garantir que não houve regressões

### Como validar
- **Sempre dois passos:** primeiro estado da BD via query SQL, depois UI visual na app
- **Sempre ciclo limpo:** apagar dados de teste antes, re-uploadar, re-importar — não validar com dados antigos
- **Nunca especular** — se uma query devolver inesperado, investigar o porquê em vez de assumir que está OK

### Critério de sucesso
Um pipeline está validado quando:
- Pré-condições cumpridas
- Cada Fase devolve o esperado em SQL
- A UI mostra o resultado correcto em todos os locais relevantes (Dashboard, Estudo, Vilões, página dedicada)

## Pipeline 1 — SS upload manual (SSMatch)

### Descrição

Workflow: utilizador uploadar SSs manualmente via "+ Importar". Vision processa, extrai TM/hero/board. Cria placeholder à espera de HH. Mais tarde, ZIP com HH chega — substitui placeholder pela HH real, mantém metadados, substitui hashes anónimos por nicks reais via Vision data, cria vilões.

### Pré-condições

- [ ] Frontend e backend deployed (sem deploys pendentes em Railway)
- [ ] Vision API configurada (variável de ambiente OpenAI)
- [ ] `FRIEND_HEROES` actualizado em `backend/app/hero_names.py` se hero da SS não for um dos heroes principais
- [ ] SSs a uploadar têm o TM visível (no nome de ficheiro `peco1.png` o TM é extraído da imagem; em ficheiros como `2026-03-31_10-17_PM_$10_$20_#5775900015.png`, o TM vem do nome)
- [ ] Se vai testar matching: ZIP com HHs reais disponível para import depois

### Passos de execução

#### Etapa A — Apagar resíduos prévios (se for teste limpo)

```sql
-- Identificar entries do hero em teste:
SELECT id, file_name FROM entries 
WHERE source='screenshot' AND raw_json->>'hero' IN ('Karluz','karluz')
ORDER BY id;

-- Identificar hands com TMs do teste:
SELECT id, hand_id FROM hands 
WHERE hand_id IN ('GG-<TM1>','GG-<TM2>',...);

-- Apagar cascata:
DELETE FROM hand_villains WHERE hand_db_id IN (<lista hands>);
DELETE FROM hands WHERE hand_id IN ('GG-<TM1>',...);
DELETE FROM entries WHERE id IN (<lista entries>);
```

**villain_notes NÃO se apagam** — são incrementais via ON CONFLICT.

#### Etapa B — Upload das SSs

1. Abrir app → **+ Importar**
2. Drag-and-drop das N SSs (ex: 10 SSs do Karluz)
3. Aguardar upload concluir
4. **Aguardar Vision processar** (~60s, depende do número)

#### Etapa C — Validar Fase 1 (placeholders criados)

```sql
-- Q1: entries criadas
SELECT id, file_name, status, 
       raw_json->>'tm' AS tm,
       raw_json->>'vision_done' AS vd,
       raw_json->>'hero' AS hero,
       jsonb_array_length(COALESCE(raw_json->'players_list', '[]'::jsonb)) AS n_players,
       jsonb_array_length(COALESCE(raw_json->'board', '[]'::jsonb)) AS n_board,
       created_at
FROM entries 
WHERE source='screenshot' AND created_at >= NOW() - INTERVAL '15 minutes'
ORDER BY created_at DESC;

-- Q2: placeholders criados (join entries↔hands via TM)
SELECT e.id AS entry_id, e.raw_json->>'tm' AS tm, e.status,
       h.id AS hand_db_id, h.origin, h.tags, h.hm3_tags,
       (h.player_names->>'match_method') AS mm
FROM entries e
LEFT JOIN hands h ON h.hand_id = 'GG-' || regexp_replace(e.raw_json->>'tm', '^TM', '')
WHERE e.source='screenshot' AND e.created_at >= NOW() - INTERVAL '15 minutes'
ORDER BY e.created_at DESC;

-- Q3: counter SSMatch
SELECT COUNT(*) AS ss_match_pending FROM hands 
WHERE origin='ss_upload' AND (player_names->>'match_method') = 'discord_placeholder_no_hh';
```

**Esperado Fase 1:**
- Q1: N entries com `vd=true`, `hero=<hero>`, `tm` preenchido, `n_players>0`, `status='resolved'`
- Q2: N hands criadas com `origin='ss_upload'`, `tags=['SSMatch']`, `hm3_tags=[]`, `mm='discord_placeholder_no_hh'`
- Q3: N (counter Dashboard SSMatch sobe N)

**UI Fase 1:**
- [ ] Dashboard → painel **SSMatch** mostra N
- [ ] Click no card → modal abre com lista
- [ ] Click numa linha → HandDetailPage abre com **PlaceholderView** (banner indigo "Mão sem HH ainda")
- [ ] SS clicável → fullscreen

#### Etapa D — Import HH (ZIP do friend)

1. Abrir app → **+ Importar**
2. Drag-and-drop do ZIP com as HHs
3. Aguardar conclusão (mostra "X mãos inseridas · Y ignoradas · Z screenshots matched")

#### Etapa E — Validar Fase 2 (matching + enrichment + villains)

```sql
-- Q1: estado das hands (matched ou ainda placeholder)
SELECT h.id, h.hand_id, h.origin, h.tags,
       h.played_at, h.tournament_name, h.position, h.result, h.hero_cards,
       (h.player_names->>'match_method') AS mm,
       (h.player_names->>'hero') AS pn_hero,
       (h.raw IS NOT NULL AND h.raw <> '') AS has_raw
FROM hands h
WHERE h.hand_id IN ('GG-<TM1>',...)
ORDER BY h.hand_id;

-- Q2: hashes resolvidos (devia ser zero)
SELECT h.hand_id, 
       (SELECT array_agg(k) FROM jsonb_object_keys(h.all_players_actions) k 
        WHERE k != '_meta' AND k ~ '^[0-9a-f]{8}$') AS hashes_remaining
FROM hands h
WHERE h.hand_id IN ('GG-<TM1>',...)
ORDER BY h.hand_id;

-- Q3: villains criados (CRÍTICO)
SELECT h.hand_id, COUNT(hv.id) AS n_villains,
       array_agg(hv.player_name ORDER BY hv.player_name) AS villain_nicks
FROM hands h
LEFT JOIN hand_villains hv ON hv.hand_db_id = h.id
WHERE h.hand_id IN ('GG-<TM1>',...)
GROUP BY h.hand_id
ORDER BY h.hand_id;

-- Q4: villain_notes incrementos
SELECT COUNT(*) AS total, 
       (SELECT COUNT(*) FROM villain_notes 
        WHERE updated_at >= NOW() - INTERVAL '20 minutes') AS recent;

-- Q5: counters Dashboard pós-import
SELECT 
  (SELECT COUNT(*) FROM hands WHERE origin='ss_upload' 
    AND (player_names->>'match_method') = 'discord_placeholder_no_hh') AS ss_match_pending,
  (SELECT COUNT(*) FROM hands WHERE origin='ss_upload' 
    AND (player_names->>'match_method') = 'anchors_stack_elimination_v2') AS ss_upload_matched;
```

**Esperado Fase 2 (com K hands matchadas, N-K placeholders):**
- Q1: K hands com `mm='anchors_stack_elimination_v2'`, `has_raw=true`, todos os campos canónicos populados; (N-K) hands com `mm='discord_placeholder_no_hh'`
- Q1: `tags=['SSMatch']` e `origin='ss_upload'` preservados em todas
- Q2: zero hashes (`hashes_remaining=None`)
- Q3: cada hand tem N>=0 villains (depende da regra B: showdown OR has_vpip não-fold)
- Q4: villain_notes recent>0 (incrementos)
- Q5: ss_match_pending=N-K, ss_upload_matched=K

**UI Fase 2:**
- [ ] Dashboard SSMatch counter desceu de N para N-K
- [ ] HandDetailPage de uma hand matched mostra **detalhe completo** (NÃO PlaceholderView)
- [ ] Hand visível em Estudo > Torneios (cumpre study_view)
- [ ] Hand visível em Vilões (se regra B/C)
- [ ] Vilões novos aparecem na lista de Vilões

### Troubleshooting comum SSMatch

| Sintoma | Causa provável | Verificar |
|---|---|---|
| `hero` ≠ esperado | hero não está em FRIEND_HEROES nem HERO_NAMES | `backend/app/hero_names.py` |
| `vision_done=false` 5min depois | Vision API timeout/erro | Logs Railway, retry endpoint |
| Placeholder NÃO criado | `_create_placeholder_if_needed` rejeitou source | Verificar source da entry, fix `aa7145f` |
| Match não acontece após import ZIP | `_insert_hand` ignora placeholder | Fix `72ff4fb` deployed? |
| Hashes não substituídos | Auto-rematch não corre para resolved | Fix `760f81a` deployed? |
| Zero villains após match | `_enrich` não chama villain creation | Fix `c3e8ece` deployed? |
| `AttributeError` em villain creation | preflop como list vs string | Fix `6562391` deployed? |

## Pipeline 2 — Discord

### Descrição

Workflow: bot Discord captura mensagens em canais subscritos (links replayer GG, imagens partilhadas). Cria entry com `source='discord'`. Vision processa para extrair hero/TM/board. Se já há HH em BD → enrich directo. Se não → cria placeholder com `origin='discord'` + `discord_tags=[<canal>]`. Quando HH chegar (HM3 ou ZIP), placeholder é substituído.

### Pré-condições

- [ ] Bot Discord ligado e subscrito aos canais correctos
- [ ] Token Discord válido em `DISCORD_BOT_TOKEN`
- [ ] Canais Discord configurados em código ou env (`DISCORD_CHANNELS`)
- [ ] Vision API configurada
- [ ] `HERO_NAMES_BY_SITE` actualizado com nicks GGPoker do utilizador

### Passos de execução

#### Etapa A — Apagar resíduos prévios (se teste limpo)

```sql
-- Identificar entries Discord do dia em teste:
SELECT id, file_name, discord_channel, discord_posted_at 
FROM entries 
WHERE source='discord' AND discord_posted_at >= '<data>'
ORDER BY id;

-- Apagar:
DELETE FROM hand_villains WHERE hand_db_id IN (
  SELECT id FROM hands WHERE entry_id IN (<lista entries>)
);
DELETE FROM hands WHERE entry_id IN (<lista entries>);
DELETE FROM entries WHERE id IN (<lista entries>);
```

#### Etapa B — Trigger captura Discord

Opções:
1. **Aguardar bot** — utilizador partilha replayer/imagem em canal subscrito; bot captura automaticamente
2. **Sync manual** — endpoint `POST /api/discord/sync` (admin) para puxar últimas N mensagens
3. **Backfill data específica** — endpoint `POST /api/discord/backfill?since=YYYY-MM-DD`

#### Etapa C — Validar Fase 1 (entry + placeholder Discord)

```sql

       raw_json->>'tm' AS tm,

-- Q2: placeholders Discord criados
SELECT e.id AS entry_id, e.discord_channel, e.raw_json->>'tm' AS tm, e.status,
       h.id AS hand_db_id, h.origin, h.tags, h.hm3_tags, h.discord_tags,
       h.screenshot_url,
       (h.player_names->>'match_method') AS mm

LEFT JOIN hands h ON h.hand_id = 'GG-' || regexp_replace(e.raw_json->>'tm', '^TM', '')

ORDER BY e.created_at DESC;

-- Q3: counter SEM MATCH (deve incluir placeholders Discord novos)
SELECT 
  (SELECT COUNT(*) FROM hands WHERE 'GGDiscord' = ANY(hm3_tags)) AS gg_discord_placeholders,
  (SELECT COUNT(*) FROM entries
```

**Esperado Fase 1:**

- [ ] Dashboard → painel **SEM MATCH** sobe N

## Pipeline 3 — HM3 .bat

### Descrição

Workflow: utilizador corre `.bat` no PC que lê BD HM3 (Holdem Manager 3). Script extrai HHs com tags HM3 + filtros (últimos N dias). Faz POST para `/api/hm3/import` com payload CSV. Backend insere mãos com `origin='hm3'`, parse de raw text, extracção de campos canonicos, criação de vilões directa (sem precisar de SS — nicks são reais nos HHs WPN/Winamax/PokerStars).

**Importante:** HM3 NÃO trás GGPoker (só sites com nicks reais).

### Pré-condições

- [ ] HM3 instalado e funcional no PC
- [ ] BD HM3 acessível
- [ ] Script `.bat` configurado com URL backend correcto
- [ ] Auth do utilizador no .bat (cookie/token)
- [ ] Tags HM3 configuradas (ex: `nota_solid`, `pos_pko`, etc.)

### Passos de execução

#### Etapa A — Apagar resíduos prévios (se teste limpo)

```sql
-- Identificar mãos HM3 do dia em teste:
SELECT id, hand_id, site, played_at, hm3_tags
FROM hands 
WHERE origin='hm3' AND played_at >= '<data>'
ORDER BY played_at DESC;

-- Apagar:
DELETE FROM hand_villains WHERE hand_db_id IN (<lista hands>);
DELETE FROM hands WHERE id IN (<lista hands>);
```

#### Etapa B — Correr o `.bat`

1. Abrir terminal Windows na pasta do .bat
2. Executar `hm3_export.bat` (ou nome equivalente)
3. Script lê HM3, gera CSV, faz POST
4. Aguardar resposta (output mostra "X mãos inseridas, Y ignoradas, Z villains criados")

#### Etapa C — Validar Fase 1 (mãos inseridas)

```sql
-- Q1: mãos HM3 recentes
SELECT id, hand_id, site, origin, hm3_tags, tags,
       played_at, tournament_name, tournament_number, buy_in,
       position, result, hero_cards,
       (player_names->>'match_method') AS mm,
       (raw IS NOT NULL AND raw <> '') AS has_raw,
       LENGTH(raw) AS raw_len
FROM hands 
WHERE origin='hm3' AND created_at >= NOW() - INTERVAL '30 minutes'
ORDER BY created_at DESC
LIMIT 50;

-- Q2: distribuição por site
SELECT site, COUNT(*) AS n
FROM hands 
WHERE origin='hm3' AND created_at >= NOW() - INTERVAL '30 minutes'
GROUP BY site
ORDER BY site;

-- Q3: distribuição por hm3_tags (multi-tag uma mão = múltiplas linhas)
SELECT unnest(hm3_tags) AS tag, COUNT(*) AS n
FROM hands 
WHERE origin='hm3' AND created_at >= NOW() - INTERVAL '30 minutes'
GROUP BY tag
ORDER BY n DESC;

-- Q4: nenhuma mão GG (HM3 não trás GG)
SELECT COUNT(*) FROM hands 
WHERE origin='hm3' AND site='GGPoker' AND created_at >= NOW() - INTERVAL '30 minutes';
-- Esperado: 0
```

**Esperado Fase 1:**
- Q1: N mãos com `origin='hm3'`, todos os campos canónicos populados (`played_at`, `tournament_name`, `position`, `result`, `hero_cards`), `has_raw=true`
- Q2: distribuição realista (Winamax + PokerStars + WPN)
- Q3: tags HM3 esperadas (não tags Discord)
- Q4: zero — HM3 nunca trás GG

**Esperado Fase 2:**
- Q1: mãos com tag `nota*` ou showdown têm villains (regra A∨B); outras têm 0
- Q2: villain_notes recent>0 (upserts incrementam hands_seen)
- Q3: zero — heroes/friends nunca devem ser villains

#### Etapa E — Auto-rematch retroactivo (HH→Discord SSs)

Se houver SSs Discord órfãs que matcham TM com mãos HM3 importadas (cenário: replayer Discord apareceu antes de bater HM3), auto-rematch deve correr. Mas como **HM3 não trás GG e Discord só recebe GG**, este cenário é raro. Pode ignorar a menos que estejas a debug situação específica.

### Troubleshooting comum HM3

| Sintoma | Causa provável | Verificar |
|---|---|---|
| 0 mãos inseridas | Filtro de dias muito apertado, BD HM3 vazia | Output do .bat |
| `played_at` NULL | Parser falhou no header | Ver linhas raw na BD HM3 |
| `position` errada | Botão dealer detectado mal | Função `_compute_positions_v2` |
| `result` calculado errado | Pot/winnings parse falhou | Função `_compute_result_bb` |
| Villains não criados | `_create_hand_villains_hm3` rebenta silenciosamente | Logs Railway |
| Hero aparece como villain | HERO_NAMES_BY_SITE não tem o nick | `backend/app/hero_names.py` |
| Mãos GG aparecem | HM3 export configurado errado | Filter no .bat |

## Pipeline 4 — Anexos Imagem ↔ Mão (Bucket 1)

### Descrição

Workflow: imagens directas Discord (anexos `.png`/`.jpg`/`.webp` ou links Gyazo) **NÃO criam mãos**. Vão como entries `entry_type='image'` para a inbox. Worker `_compute_match_candidates` cruza estas entries com mãos existentes via janela ±90s (regra de produto em `CLAUDE.md` "Imagens de contexto Discord"). Match primário: outra entry Discord no mesmo canal ±90s que tenha hand sibling. Match fallback: hand com `origin IN ('hm3','hh_import')` ±90s (qualquer canal). Resultado: row em `hand_attachments` ligando entry image ↔ hand. UI mostra ícone `📎 N` na lista (`HandRow.jsx`) + secção CONTEXTO no detalhe (`HandDetailPage.jsx`) com thumbnail 200px. Detalhes técnicos completos no MAPA §2.11.

### Pré-condições

- [ ] Tabela `hand_attachments` criada via `ensure_hand_attachments_schema()` (corre no lifespan do FastAPI)
- [ ] Endpoint `/api/attachments/{preview,match}` registado em `main.py`
- [ ] Triggers retroactivos activos em `sync_and_process` (`discord.py:147`) e `import_hm3` (`hm3.py:1205`) — ambos via `asyncio.create_task`
- [ ] Pelo menos 1 entry com `source='discord' AND entry_type='image'` em BD para testar
- [ ] Hand candidata: ou (a) entry replayer_link no mesmo canal ±90s com hand associada, ou (b) hand `origin IN ('hm3','hh_import')` com `played_at` ±90s

### Passos de execução

#### Etapa A — Cleanup (se teste limpo)

Padrão genérico em Pipeline 1 Etapa A. Específico para anexos imagem:

```sql
-- Identificar attachments do teste:
SELECT id, hand_db_id, entry_id, channel_name, posted_at
FROM hand_attachments
WHERE created_at >= '<data>'
ORDER BY id;

-- Apagar attachments de teste (NÃO afecta entries — entry_id fica SET NULL):
DELETE FROM hand_attachments WHERE id IN (<lista>);
```

**Não tocar em `entries.status`** — o worker não escreve nessa coluna (ver MAPA §2.11 Armadilha 4 / fix `b9f0753`). O estado "anexada" é representado APENAS pela existência de row em `hand_attachments`. Se quiseres re-correr o worker, basta apagar a row em `hand_attachments` — o filtro `_pending_image_entries` usa `NOT IN (hand_attachments)`.

Para usar a flag idempotente do script de backfill: `python backfill_attach_orphan_images.py --rollback` (apaga rows das 3 entries-alvo conhecidas 13/17/87).

#### Etapa B — Trigger

3 opções:

1. **Sync Discord** — `POST /api/discord/sync-and-process`. Após o backfill GGDiscord, dispara `asyncio.create_task` que corre `run_match_worker(limit=100)`. Cobre o caso "imagem chegou no mesmo sync que a HH/replayer sibling".
2. **Import HM3** — `POST /api/hm3/import` (via `.bat` ou UI). Após o auto-rematch SS órfãs, dispara `asyncio.create_task`. Cobre o caso "mão HM3 acabada de inserir tem imagem Discord órfã ±90s".
3. **Endpoint manual** — `POST /api/attachments/match?confirm=true&limit=N` (admin). Útil para forçar reprocessamento sem aguardar Discord/HM3. `GET /api/attachments/preview?limit=N` é o dry-run.

#### Etapa C — Validar Fase 1 (entries image em pending)

```sql
-- Q1: entries image candidatas (universo do worker)
SELECT id, raw_text, discord_channel, discord_posted_at, status
FROM entries
WHERE source = 'discord' AND entry_type = 'image'
  AND id NOT IN (SELECT entry_id FROM hand_attachments WHERE entry_id IS NOT NULL)
  AND discord_posted_at IS NOT NULL
ORDER BY discord_posted_at ASC;

-- Q2: hands sibling candidatas (mesmo canal, ±90s)
SELECT e2.id AS sibling_entry_id, e2.discord_channel,
       h.id AS hand_db_id, h.hand_id, h.played_at,
       ABS(EXTRACT(EPOCH FROM (e2.discord_posted_at - <img.posted_at>::timestamptz)))::int AS sibling_delta_s,
       ABS(EXTRACT(EPOCH FROM (h.played_at - <img.posted_at>::timestamptz)))::int AS img_to_played_s
FROM entries e2
JOIN hands h ON h.entry_id = e2.id
WHERE e2.discord_channel = '<canal_da_img>'
  AND ABS(EXTRACT(EPOCH FROM (e2.discord_posted_at - <img.posted_at>::timestamptz))) <= 90
  AND h.played_at >= '2026-01-01'
ORDER BY sibling_delta_s ASC;

-- Q3: estado pós-trigger
SELECT id, hand_db_id, entry_id, channel_name, match_method, delta_seconds,
       img_b64 IS NOT NULL AS has_b64, mime_type, posted_at
FROM hand_attachments
WHERE entry_id IN (<lista entries de teste>)
ORDER BY id;
```

**Esperado:**
- Q1: N entries pendentes ANTES do trigger.
- Q2: ≥1 candidate por imagem testada; tiebreaker `delta_s ASC, played_at ASC, h.id ASC`.
- Q3: K rows após trigger (K = número de imagens com candidate). `match_method='discord_channel_temporal'` (primário) ou `'hm3_temporal_fallback'`. `delta_seconds = |posted_at - hand.played_at|` (não sibling delta — ver MAPA §2.11 detalhes técnicos).

**UI Fase 1:**
- [ ] `GET /api/hands/{id}` para hand sibling devolve `attachments: [...]` com K items.
- [ ] `GET /api/hands` lista a hand com `attachment_count = K`.
- [ ] `HandRow.jsx` mostra `📎 K` na linha de botões (apenas para mãos matched — ver Etapa D edge case).
- [ ] `HandDetailPage.jsx` mostra secção CONTEXTO entre HERO+BOARD e MESA com thumbnail 200px clicável.

#### Etapa D — Cenários edge

**(D.1) Imagem ±90s de DUAS mãos no mesmo canal.** Caso real pré-Bucket 1 (entry 13 com 2 candidates: hand 117 sibling 7s, hand 85 sibling 64s). Tiebreaker: `delta_s ASC, played_at ASC, h.id ASC` — escolhe hand mais próxima temporalmente; empate exacto → mais antiga. Anexa a UMA só. Verificar:

```sql
-- Confirmar que o worker escolheu a mão certa para a imagem com 2 candidates:
SELECT entry_id, hand_db_id, delta_seconds
FROM hand_attachments
WHERE entry_id = <entry_id_com_2_candidates>;
-- Esperado: 1 row, hand_db_id da mais próxima.
```

**(D.2) Cross-post múltiplo da mesma URL Gyazo em N canais.** Caso real (entries 7/46/133 com mesma URL `c0871ebc6529...` em 3 canais diferentes, posted_at 3-6s entre si). Decisão Q3 SPEC: **N/A** — dedupe não implementado. Cada cross-post é entry independente; cada uma tenta match no seu próprio canal. Se nenhum tiver hand sibling no mesmo canal ±90s, todas ficam órfãs (esperado). Comportamento observado em prod 2026-04-26: 3 cross-posts, 3 órfãs. Aceitável.

#### Etapa E — Validar match retroactivo HH→imagem

Quando HH chega via HM3 ou ZIP **depois** da imagem Discord ter sido sincronizada, o trigger retroactivo (`asyncio.create_task` em `import_hm3` / `sync_and_process`) deve reanexar via match fallback. Sequência típica:

1. Discord sync → entry image criada com `discord_posted_at = T`.
2. Worker corre primeira vez → sem hand candidate ±90s → entry fica órfã.
3. Mais tarde, HM3 import insere hand com `played_at` ±90s de T.
4. Trigger retroactivo dispara após import → `_compute_match_candidates` re-corre → match fallback (`origin='hm3'`) → INSERT em `hand_attachments`.

Validar:
```sql
-- Antes do import HM3:
SELECT COUNT(*) FROM hand_attachments WHERE entry_id = <entry_image_id>;
-- Esperado: 0

-- Após import HM3 + ~1s para asyncio.create_task correr:
SELECT id, hand_db_id, match_method, delta_seconds
FROM hand_attachments WHERE entry_id = <entry_image_id>;
-- Esperado: 1 row, match_method='hm3_temporal_fallback'
```

**Subtileza crítica** (MAPA §2.11 Armadilha 1): se a hand candidate é placeholder Discord que vai virar matched (HH chega), o `_insert_hand` apaga o placeholder (`ON DELETE CASCADE` apaga attachments associados), re-insere com novo id, e o trigger reanexa. `attachment_count` cai para 0 momentaneamente. Gap visual: milissegundos. Não interpretar como bug se for visto logo após import.

## Lições da sessão de validação 24 Abril 2026

### Boas práticas

1. **Validação por Fase, não tudo de uma vez.** Para SSMatch, separámos Fase 1 (placeholders) de Fase 2 (matching+enrichment+villains). Cada Fase com queries específicas. Isso isolou bugs.

2. **Ciclos limpos > backfills complicados.** Apagar dados de teste e re-uploadar é sempre mais limpo que tentar corrigir estado parcial. 5 ciclos foram necessários para SSMatch — todos rápidos.

3. **Queries SQL antes da UI.** A BD não mente; a UI pode ter cache, race conditions, render bugs. Sempre validar BD primeiro.

4. **Listar critérios de sucesso ANTES de testar.** Saber exactamente o que esperar. Ajuda a detectar desvios subtis (ex: tag preservada vs perdida).

### Anti-patterns descobertos

1. **`try/except` silenciosos escondem bugs.** Bug #6 e #7 do SSMatch ficaram escondidos por `except Exception` no enrich. Lição: logger.error com tracebacks completos, ou re-raise em desenvolvimento.

2. **Formatos diferentes para o mesmo campo.** `actions['preflop']` é list no parser GG bulk, string no parser MTT. Convergir formatos ou normalizar defensivamente em helpers partilhados.

3. **Não testar o caminho "menos provável".** Pipeline GG anonimizado (Hero literal) tem comportamento subtilmente diferente de pipeline com nicks reais. Sempre testar variantes.
