## 1. #IRE-MB — constante do bounty derivada (commit `942ec08`)

**Problema:** a constante `0.25` do IRE estava hardcoded (W3cray table + `_formula_fallback`),
tratando todos os formatos PKO como 50/50. Big Bounty GG (30/70) mostrava IRE errado
contra a tabela errada, apresentado como válido.

**Fórmula:** `constante = KOP_fraction × instant_fraction`, onde
`KOP_fraction = buy_in_bounty / (buy_in_entry + buy_in_bounty)` (rake fora) e
`instant_fraction = 0.5` (validado empiricamente em 2 pontos: 50/50→0.25, 30/70→0.35).

**Sequência T0→T6 (validada passo-a-passo pelo Web):**

- **T0** — `test_ire.py` novo (14 casos): congelar o comportamento legacy 0.25 ANTES
  do refactor (baseline anti-regressão). `ire.py` não tinha cobertura directa.
- **T1** — parametrizar a constante (`_formula_fallback`/`lookup_ire_pct` ganham
  `constant=0.25` default). Zero mudança de comportamento.
- **T2** — `derive_kop_fraction` (GG via TS / Winamax+WPN None / PS via header raw_hh)
  + `derive_constant`. Helpers puros.
- **T3** — wiring: `hands.py` (`list_hands`+`get_hand`) faz LEFT JOIN a
  `tournament_summaries` para trazer `buy_in_entry`/`buy_in_bounty`; `compute_ire`
  deriva e propaga a constante.
- **T4** — docstrings: Super KO continua **escondido** (decisão b — falta validação
  empírica do `instant_fraction` Super KO).
- **T5** — guarda **PKO-only**: Mystery KO mantém 0.25 legacy (bounty aleatório → a
  fórmula de bounty fixo não aplica; suporte real é #MYSTERY-KO-DUAL-SUPPORT).
- **T6** — testes finais (monotonia, banda, edge cases). Total 30.

**Decisões product (Web):**
- **α** — banda `±0.01` à volta de 0.25: o split GG 50/50 real dá ~0.247 (rake);
  a banda absorve esse ruído → PKO standard continua na tabela calibrada (sem mudança
  de valores). Só Big Bounty (0.35) / formatos genuinamente diferentes caem na fórmula.
- **Ramo PS dormante:** `derive_kop_fraction` suporta PS (header `[A=PP, B=KOP, C=rake]`,
  ordem ≠ GG TS `[PP, rake, KOP]`), mas `compute_ire` está gated a GGPoker → o ramo PS
  fica construído+testado mas não flui até o gate de site ser relaxado (fora de âmbito).

**Correcção a erro do Web:** o diff inicial importava `_PS_3COMP_AMOUNTS_RE` de
`app.services.tournament_format`; o ficheiro vive em **`app.utils.tournament_format`**
(grep confirmou — só lá existe). Mantido `app.utils`.

**Correcção a teste do Web (T3):** a asserção `approx(7.446808)` partia porque
`compute_ire` arredonda `ire_pct` a 1 casa (`round(.,1)` → 7.4). Ajustada para
`approx(7.4, abs=0.05)` + asserção separada do valor cru ao nível de `_formula_fallback`.

**User-facing:** só muda nos **Big Bounty GG com TS importado** (IRE correcto, mais alto).
PKO standard, Mystery KO, Super KO, sem-TS e não-GG: byte-a-byte iguais.

## 5. Info técnica nova do Rui — Mystery KO (reforça #MYSTERY-KO-DUAL-SUPPORT)

O Mystery KO **não é progressivo**: `instant_fraction = 1.0` (todo o bounty é cash
imediato, não há "head" a acumular). Os bounties só ligam **no ITM** (ou, mais raramente,
no fecho do late registration), e o valor de cada um é por **sorteio** (random até ao KO).
Implicações para o futuro #MYSTERY-KO-DUAL-SUPPORT:
- O IRE para Mystery não pode usar bounty fixo — precisa de EV/média do pool de bounties
  restantes, e só faz sentido depois do threshold de desbloqueio (ITM).
- Confirma a decisão pt43 de **excluir Mystery do âmbito #IRE-MB** (guarda PKO-only, T5).
