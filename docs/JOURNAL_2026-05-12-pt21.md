## Trabalho realizado

### Bloco 1 — G3 (tabela hrc_jobs schema, commit `5b9c10a`)

**Motivação:** sem persistência por mão, o estado (submetido / a correr / done / failed) é opaco. G2 (POST /results) escreve aqui; G6 (badge UI) lê aqui.

**Decisões fechadas (S1-S7):**

| # | Decisão | Valor |
|---|---|---|
| S1 | `hand_db_id` é FK real? | **Sim**, `REFERENCES hands(id) ON DELETE CASCADE`. NB: `hands.id` é `SERIAL` (INTEGER), portanto `hand_db_id INTEGER` (plano original dizia BIGINT, ajuste de tipo). |
| S2 | `result_zip` URL vs BYTEA? | **BYTEA + result_zip_size INTEGER**. URL+storage externo fica como tech debt `#HRC-RESULT-STORAGE-MIGRATION`. |
| S3 | `status` enum vs TEXT? | TEXT com `CHECK (status IN ('submitted','running','done','failed','expired'))`. |
| S4 | `meta_json` JSONB vs TEXT? | **JSONB**. |
| S5 | PK 1 job por mão? | `id BIGSERIAL` + `UNIQUE (hand_db_id)`. Histórico via tech debt `#HRC-JOBS-HISTORY-SUBSEQUENT` quando regra de produto exigir. |
| S6 | Índice `(status, submitted_at)`? | **Sim** desde MVP. |
| S7 | `ensure_hrc_jobs_schema()` em ficheiro novo? | `backend/app/services/hrc_jobs.py` novo. |

**Schema final:**

```sql
CREATE TABLE IF NOT EXISTS hrc_jobs (
    id              BIGSERIAL PRIMARY KEY,
    hand_db_id      INTEGER NOT NULL REFERENCES hands(id) ON DELETE CASCADE,
    status          TEXT NOT NULL DEFAULT 'submitted'
                    CHECK (status IN ('submitted','running','done','failed','expired')),
    submitted_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    completed_at    TIMESTAMPTZ,
    result_zip      BYTEA,
    result_zip_size INTEGER,
    meta_json       JSONB,
    error           TEXT,
    UNIQUE (hand_db_id)
);
CREATE INDEX IF NOT EXISTS idx_hrc_jobs_status_submitted_at
    ON hrc_jobs (status, submitted_at);
```

**Verificação BD prod:** confirmada via `railway run python` — tabela + 3 índices (`pkey`, `hand_db_id_key`, `idx_hrc_jobs_status_submitted_at`) + 7 constraints (FK CASCADE, UNIQUE, PRIMARY KEY, CHECK status, 4 NOT NULLs).

**Helper novo** `require_auth_or_api_key` em `app/auth.py` (~45 linhas):

- Caminho 3: nenhum → 401 `"Não autenticado"`.

Aplicado em `GET /api/queue/hrc` (1 linha de troca em `routers/queue.py`).

| D-G2-EXTRA-4 | WARNING no `failed` com file? | **Sim** desde já |

**Pipeline:**

```
POST /api/queue/hrc/results?hand_id=GG-...&status=done
  -> auth: cookie OU Bearer (G4)
  -> query SELECT id FROM hands WHERE hand_id = ?  -> 404 ausente
  -> file.read() <= 50 MB                          -> 413 oversized
  -> extract_meta_from_result_zip                  -> 400 invalid zip / sem meta
  -> meta_aug = {**meta, hand_id, received_at, received_from}
  -> upsert_hrc_job_result -> INSERT ... ON CONFLICT (hand_db_id) DO UPDATE
  -> log INFO hrc_jobs: upsert ...
  -> 200 {hand_db_id, status, action, meta}
```

**Helpers novos** em `services/hrc_jobs.py`:
- `extract_meta_from_result_zip(bytes) -> dict` — valida + parsea meta.json (raise `ValueError`).
- `upsert_hrc_job_result(*, hand_db_id, status, result_zip, meta_json, error) -> {id, inserted}` — UPSERT SQL com `(xmax = 0) AS inserted`; `submitted_at` NÃO está em `DO UPDATE SET`.

**Tests:** 11 cases novos. Slowest: 0.01s (bem abaixo do limite 2s).

### Bloco 6 — 2 achados operacionais (durante verificação BD em G3)

1. **`DATABASE_PUBLIC_URL` no serviço Postgres stale.** Password (32 chars) não bate com a real do `poker-app` (31 chars). App usa internal URL, portanto não bloqueia prod. Workaround para queries externas: usar password do `poker-app` + proxy público do `Postgres`. Tech debt pequeno (não inventoriado formalmente — re-abrir como `#RAILWAY-POSTGRES-PUBLIC-URL-STALE` se virar reincidente).

2. **`backend/.env` local com encoding não-UTF8.** Causa `UnicodeDecodeError 0xe3` em scripts que importam `app.db` (`load_dotenv()` carrega bytes Latin-1 e contamina env). Não afecta prod nem tests. Workaround: ler vars via `subprocess(railway variables --kv)` em vez de `app.db.get_conn`. Tech debt pequeno (idem — `#LOCAL-ENV-ENCODING` se virar reincidente).
